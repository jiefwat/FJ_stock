# 主题优先研究工作台设计

## 目标

当前四个模块已经有真实数据和完整清单，但用户仍需从指数、财务事实和候选列表中自行理解重点。本次改造固定回答四个问题：现在是什么结论、主导主题是什么、内部如何分化、下一步看什么。

统一阅读顺序为：

```text
一句话结论 -> 主题与分化 -> 行动和失效条件 -> 证据
```

首页不增加方法旁白。完整性通过结构化主题区、真实市场宽度、持仓分化和证据抽屉实现。

## 方案选择

### 方案 A：主题优先决策台（采用）

扩展现有供应商中性协议，新增模块分区和决策标签；大盘增加市场宽度、热门股能力，持仓增加行业主题能力。页面按模块解释同一份结构化结果。

优点是用户能立即看到主题和结论，且不产生第二套分析系统。缺点是研究引擎需要增加模块级合成逻辑。

### 方案 B：恢复旧行情仪表盘

旧系统有更多图表，但会重新引入大量指标、数据质量面板和解释话术，不能解决“一眼看不到重点”，不采用。

### 方案 C：以对话替代页面

对话适合追问，不适合每天固定扫大盘、持仓和机会，也无法保证四个模块信息一致，不采用。

## 产品协议

保持 `verdict / action / primary_risk / findings / details / module_items` 兼容，新增：

```text
decision_label: 修复中 / 偏强 / 偏弱 / 先处理风险 / 等待确认 / 有主线
module_sections[]:
  key: market-themes / market-breadth / market-hot / portfolio-themes /
       portfolio-divergence / stock-thesis / opportunity-themes / opportunity-candidates
  title: 用户可读标题
  conclusion: 该分区的一句话解释
  items[]: 继续复用 ResearchModuleItem
```

`module_sections` 只能包含产品字段，不包含能力 id、供应商、trace、网关或凭证。

## 分析方法

### 每日大盘

新增两个逻辑能力，底层仍使用官方 A 股筛选能力：

- `breadth`：同花顺全 A 的上涨、下跌、平盘、涨停和跌停家数。
- `hot_stock`：成交额前 10 股票的代码、名称、涨跌幅、成交额、所属概念和行业。

大盘结论规则：

1. 上涨家数显著多于下跌家数且指数趋势不弱，标记“偏强”。
2. 当日上涨但 5 日区间仍负，标记“修复中”。
3. 下跌家数显著更多或跌停扩散，标记“偏弱”。
4. 宽度缺失时只输出指数条件判断，明确“分布待补”。

主题区展示前 5 个行业/概念，每项保留热度、涨跌或成交事实。热门股必须显示所属主题；没有主题的股票标记“主题待确认”，不硬归类。

页面顺序：市场结论、主题雷达、涨跌分布、热门股票、指数证据、完整依据。

### 我的持仓

每只持仓由事件、机构预期、行情资金扩展为行业主题四类证据，仍最多分析 20 只。主题聚合只使用代码和名称查询得到的公开行业/概念信息。

组合结论规则：

1. 先统计主题暴露的股票数量，不使用股数、成本或权重推断仓位。
2. 同主题内按近期涨跌、事件和预期分为“相对强”“相对弱”“待确认”。
3. 负面事件或预期下修优先进入“今天先处理”。
4. 主题只有一只持仓时只说明暴露，不虚构内部分化。

页面顺序：组合结论、主题暴露、主题内部分化、今天先处理、全部持仓、完整依据。

### 个股分析

八维证据保持不变，但先合成 `decision_label`：

- 财务或事件存在明确负面信号：`基本面承压`。
- 经营/预期改善但价格或报告证据不足：`等待确认`。
- 六维以上可用且没有明确负面信号：`可继续跟踪`。
- 不足六维：`证据不足`。

一句话结论必须包含“现状 + 原因 + 下一步”，例如：`基本面承压：利润和现金流仍弱，先等业绩转正和价格确认。` 页面随后展示确认条件、推翻条件和八维证据，不展示方法论术语。

页面顺序：个股结论、为什么、确认条件、失效条件、八维覆盖、完整依据。

### 热点机会

机会页用行业方向和候选数据形成两层结论：先判断今天有没有可持续主线，再按主题组织候选。

- 主题必须包含强度或成交证据。
- 候选必须有代码、名称、主题、入选依据和淘汰条件。
- 只有热度、没有业绩/成交证据的候选标记“待确认”。
- 负面事件只用于剔除或降级，不能作为推荐理由。

页面顺序：机会结论、今日主题、主题内候选、确认条件、风险排除、完整依据。

## 页面设计

延续现有纸白、墨蓝、克制绿红的金融终端语言，不更换整体品牌。唯一新的视觉记忆点是“主题雷达带”：紧跟结论的横向主题卡，每张只显示主题名、状态和一条证据。

桌面：

```text
┌ 结论标签 ─ 一句话结论 ─────────────┐
│ 现在怎么做                  最大风险 │
├ 主题雷达：主题 A | 主题 B | 主题 C ─┤
├ 模块分区：分布 / 分化 / 热门股 ─────┤
└ 证据与完整清单 ────────────────────┘
```

移动端：

```text
结论标签
一句话结论
三个主题（横向滚动）
一个行动 / 一个风险
分布或分化
完整清单
```

390px 下不得横向溢出；主题带自身可横向滚动，但页面根节点宽度必须等于视口。

## 降级与安全

- 宽度缺失：隐藏分布条，显示“市场分布待补”。
- 主题缺失：保留股票结论，显示“主题待确认”。
- 持仓行业能力失败：该持仓进入“主题待补”，不影响事件和行情分析。
- 旧新闻、公告和研报继续执行 30 天首屏门禁。
- 公网继续要求登录；持仓和个股不进入共享快照。
- 动态页面继续只用 `textContent` 和 DOM 节点，不使用 `innerHTML`。

## 验收标准

1. 大盘首屏先出现主题，展示真实上涨/下跌/平盘/涨停/跌停家数和 5–10 只热门股票。
2. 热门股票能显示所属概念或行业；缺失时明确待确认。
3. 当前 11 只持仓全部分析，并按主题聚合；同主题多只股票展示相对强弱分化。
4. 个股首屏出现用户可理解的决策标签和“现状 + 原因 + 下一步”结论。
5. 机会页先显示主题结论，再按主题展示候选及淘汰条件。
6. 四页均保持结论、行动、风险、证据日期和缺口。
7. 1280px 与 390px 无页面级横向溢出，移动行动坞不遮挡内容。
8. 专项测试、真实接口、浏览器和公网验证通过；本地、GitHub、服务器 `main` 一致。
