# StockTS 专业多视角股票研究系统设计

## 1. 决策

StockTS 从“统一模板下的多模块展示”升级为“统一证据协议下的四种研究方法”。每日大盘、我的持仓、个股分析和热点机会共享事实、推断、未知、反证、确认与失效六类证据状态，但各自采用不同的分析维度、比较基准和最终输出。

问财官方能力负责提供可核查事实，StockTS 负责数据质量门禁、时间对齐、横向比较、因果链、矛盾识别和最终研究结论。任何外部能力都不能直接决定仓位、机会排序或买卖动作。

## 2. 当前问题

当前项目已经接入财务、经营、机构预期、事件、行情资金、行业、公告和研报等能力，但产品层主要执行“并行查询 -> 抽取少量字段 -> 压缩成通用发现”。这会产生四类问题：

1. 四个工作台的分析框架过于相似，没有体现市场事实、组合风险、公司研究和条件预测的差异。
2. 外部能力返回事实后缺少趋势变化、同行比较、预期差、因果关系和反证合成。
3. 缺失维度仍可能显示分数，覆盖率与评分含义冲突。
4. 个股页没有一个用户可主动触发、可看到进度、可专项追问的深度研究入口。

## 3. 能力来源与使用边界

### 3.1 官方事实能力

StockTS 使用以下问财官方能力构建事实层：

| 研究域 | 官方能力 | 事实用途 |
| --- | --- | --- |
| 基本资料 | `hithink-basicinfo-query` | 上市时间、公司类型、发行主体和基础身份 |
| 公司经营 | `hithink-business-query` | 主营构成、客户供应商、重大合同、参控股公司 |
| 财务质量 | `hithink-finance-query` | 营收、利润、ROE、毛利率、负债率和现金流 |
| 股东治理 | `hithink-management-query` | 股本、股东户数、实控人、质押和重要持有人 |
| 行业位置 | `hithink-industry-query` | 行业估值、盈利、行情和同行排名 |
| 机构预期 | `hithink-insresearch-query` | 盈利预测、评级、目标价和一致预期 |
| 行情资金 | `hithink-market-query` | 价格、成交量、资金流、大小单和技术指标 |
| 公司事件 | `hithink-event-query` | 业绩预告、增发、质押、解禁、调研和监管函 |
| 公告核查 | `announcement-search` | 财报、分红、回购、增持和重组原文线索 |
| 研报观点 | `report-search` | 投资逻辑、评级、目标价和机构分歧 |
| 新闻催化 | `news-search` | 政策、产业、公司进展和舆情风险 |
| A股筛选 | `hithink-astock-selector` | 财务、技术、行情和概念组合筛选 |
| 板块筛选 | `hithink-sector-selector` | 行业估值、资金、涨跌和板块类型筛选 |
| 指数与宏观 | `hithink-zhishu-query` / `hithink-macro-query` | 指数状态和宏观变量 |

页面和产品 JSON 不展示供应商名称、Logo、能力 ID、版本、trace、网关地址或密钥。

### 3.2 只吸收方法、不直接执行的社区框架

社区 SkillHub 中的财务报表深度解读、估值模型、盈利预期修正、行业轮动、市场情绪、多因子选股、公司事件驱动、市场微观结构和风险压力测试仅作为方法参考。未经独立验证，不把社区技能安装到生产主链，也不允许其直接生成买卖结论。

## 4. 统一专业研究协议

### 4.1 六类证据状态

每个研究维度必须输出以下状态之一：

- `fact`：带来源日期、可复核的原始事实。
- `inference`：由 StockTS 根据事实形成的确定性推断。
- `unknown`：数据缺失或过期，不参与评分。
- `counter`：与当前研究假设冲突的证据。
- `confirmation`：继续保留判断必须满足的条件。
- `invalidation`：判断失效或移出观察的条件。

缺失维度显示 `--` 和缺口说明，不显示中性分、默认分或伪造置信度。

### 4.2 研究合成顺序

每个工作台按相同顺序处理证据，但使用不同维度：

1. 数据日期、来源和覆盖率门禁。
2. 当前事实与历史变化比较。
3. 与市场、行业、同行或组合基准比较。
4. 形成“驱动 -> 结果 -> 可持续性”的因果链。
5. 提取最强支持、最大反证和未解决矛盾。
6. 给出确认条件、失效条件和下一次复核事件。

评分只能用于同一研究池内排序，不能替代证据，也不能解释为上涨概率、目标收益或仓位比例。

### 4.3 协议版本

研究快照增加 `research_contract_version`。大盘和机会快照只有在版本、事实日期、来源流水线版本和必需维度同时匹配时才能作为当日结论。部署新的研究协议后，旧版本快照必须降级为历史参考或立即重算，不能与新算法混合展示。

## 5. 四个工作台的方法

### 5.1 每日大盘：市场事实与状态识别

分析维度：

1. 宏观与政策变量。
2. 主要指数趋势和波动。
3. 全市场宽度与涨跌停结构。
4. 成交额、流动性和资金方向。
5. 大小盘、价值成长和周期防御风格。
6. 行业轮动、主题扩散和主线持续性。
7. 市场情绪、拥挤和极端波动。
8. 当日及未来一周事件日历。

输出只包含已发生事实、市场阶段、主要驱动、核心矛盾和风险状态。禁止输出个股买卖、账户仓位或“投资候选”。

### 5.2 我的持仓：组合相对风险与处理顺序

分析维度：

1. 持仓研究假设是否发生变化。
2. 财务、经营和机构预期变化。
3. 个股趋势、资金和波动结构。
4. 行业与主题暴露集中度。
5. 持仓相关性和共同风险因子。
6. 单股及组合回撤贡献。
7. 公告、事件和催化剂日历。
8. 与当前市场状态的适配度。

输出为风险排序、继续持有条件、减风险条件、禁止动作和下一复核事件。在账户总资产、现金和风险偏好缺失时，不推断绝对仓位百分比。

### 5.3 个股分析：公司研究备忘录

分析维度：

1. 公司身份与商业模式。
2. 收入结构、客户供应商和竞争优势。
3. 收入、利润和盈利能力趋势。
4. 现金流质量、资本开支和资产负债表。
5. 绝对估值、历史估值和同行估值。
6. 行业景气、竞争位置和产业链传导。
7. 机构一致预期、盈利修正和观点分歧。
8. 股东结构、治理、质押和重要持有人变化。
9. 价格趋势、成交、资金和波动。
10. 公告、事件、新闻和研报催化。
11. 当前研究假设、预期差、反证和失效条件。

输出顺序固定为：一句话研究假设、关键变化、最强支持、最大反证、预期差、估值匹配、三种情景、确认条件、失效条件和下一复核日期。

### 5.4 热点机会：条件预测与风险排除

分析维度：

1. 主题启动时间和多日持续性。
2. 板块宽度、扩散和内部强弱分化。
3. 行业景气、盈利趋势和估值位置。
4. 资金强度、成交拥挤和流动性。
5. 指数、风格和市场状态适配度。
6. 公司财务与经营质量底线。
7. 机构预期变化和市场预期差。
8. 公告、新闻、政策和事件催化。
9. 高位回撤、事件和基本面风险排除。
10. T+1、T+3、T+5 的确认与失效条件。

先筛主题，再筛股票。候选必须通过主题持续、公司质量、价格确认和风险排除四道门，不按最近单日涨幅直接入选，也不为填满 Top10 降低门槛。

## 6. 个股“深度研究补强”模块

### 6.1 位置与首屏行为

模块放在个股切换输入区之后、主结论之前，保持紧凑，不占用完整首屏。默认只展示一句说明、六个维度状态和一个主按钮，不在页面打开时自动请求外部服务。

主按钮文案为 `运行深度研究`。已有五分钟内缓存时显示 `查看最新深度研究`；研究完成后按钮变为 `更新深度研究`。

### 6.2 六个用户可理解的分组

页面把底层能力合并为六组，不展示供应商实现：

1. 公司与经营。
2. 财务与估值。
3. 行业与同行。
4. 机构与预期。
5. 资金与技术。
6. 公告与事件。

每组状态为 `等待研究 / 研究中 / 已确认 / 待补数据 / 请求失败`。部分失败不阻断其他分组。

### 6.3 结果内容

完成后模块只展示：

- 新增的关键事实，最多三条。
- 与本地判断一致的证据，最多两条。
- 与本地判断冲突的证据，最多两条。
- 最重要的数据缺口。
- 外部事实日期和本地档案日期。

完整字段放入折叠的“查看深度证据”，不重复个股页已有的价格、代码和通用风险提示。

### 6.4 专项追问

模块提供六个快捷问题：`财务质量`、`估值与同行`、`机构预期`、`资金结构`、`公告事件`、`新闻研报`。另提供一个自然语言输入框，用户可以询问“过去四个季度现金流为什么变差”等具体问题。

专项追问只调用最相关的一项官方能力。问题、股票代码和股票名称可以发送；持股数量、成本、权重、备注、账号、Cookie 和组合列表不得发送。

### 6.5 API 与缓存

新增供应商中性的登录态接口：

```text
POST /api/research/stock/deep
```

请求只接受：

```json
{
  "code": "600519",
  "name": "贵州茅台",
  "focus": "all | finance | valuation | consensus | market | event | news",
  "question": "",
  "refresh": false
}
```

响应使用 `StockDeepResearchResult`，包含研究日期、六组状态、事实、支持、反证、未知项和数据日期。服务端并行调用，单能力超时不超过现有 12 秒，总结果缓存五分钟。错误响应只使用产品语言，不回显供应商、技能、trace 或底层异常原文。

## 7. 数据流与组件边界

```text
本地行情/持仓/快照
  -> 专业研究方法层
  -> 本地基础结论
  -> 官方事实能力并行补强
  -> 能力级语义提取与日期校验
  -> 事实/推断/未知/反证分类
  -> 工作台专属研究合成
  -> 供应商中性产品 JSON
  -> 四工作台与个股深度研究模块
```

建议新增 `src/stock_ts/research_method.py`，集中定义四工作台维度、覆盖规则和确定性合成；新增 `src/stock_ts/stock_deep_research.py`，只负责个股深度研究请求、分组、缓存和供应商中性结果。`iwencai.py` 保持外部适配器职责，`research_engine.py` 保持能力调度职责，页面不直接解释供应商字段。

## 8. 错误与降级

- 未配置外部能力：本地结论照常显示，模块提示“深度数据服务尚未配置”。
- 单能力失败：对应分组显示“请求失败”，其他分组继续完成。
- 返回字段不足：显示“待补数据”，不生成默认分。
- 数据日期冲突：保留两方事实并标记冲突，不自动选择对当前结论更有利的一方。
- 请求超时：保留本地结论和上一次有效缓存，明确缓存日期。
- 所有异常：页面只显示可执行的恢复建议，不显示 Python、JSON、路径、网关或外部供应商错误。

## 9. 安全与产品边界

- 仅登录用户可以调用深度研究接口。
- 浏览器不能提交能力 ID、版本、网关或自定义服务地址。
- API Key 只存在于服务器权限受限环境。
- 外部结果不自动写持仓、不自动下单、不直接修改候选排序。
- 页面不展示问财、同花顺、SkillHub、Logo 或外部能力标识。
- 研究内容只用于个人研究和复盘，不构成投资建议。

## 10. 测试与验收

### 10.1 方法测试

- 四个工作台拥有不同的必需维度、比较基准和输出顺序。
- 缺失维度不显示分数，不进入覆盖率分子。
- 市场页面不出现仓位、买卖或个股投资候选语义。
- 持仓页不在缺少总资产和现金时生成绝对仓位。
- 个股结论必须同时包含支持、反证、预期差和失效条件。
- 机会候选必须通过主题、公司、价格和风险四道门。

### 10.2 深度研究接口测试

- 登录、请求白名单、大小限制、限流、超时和五分钟缓存。
- 六个分组可以部分成功并返回供应商中性结果。
- 专项追问只路由一项能力。
- 持仓数量、成本、账号和 Cookie 永不进入外部请求。
- 错误响应不包含供应商、能力 ID、trace、密钥或底层异常。

### 10.3 页面测试

- 个股页首屏出现紧凑的深度研究模块，但不自动发起请求。
- 一键研究显示分组进度、部分成功、缓存日期和更新入口。
- 快捷问题和自然语言追问可用，键盘焦点清晰。
- 完整证据默认折叠，1440px 和 1680px 无横向溢出。
- 深链接刷新后仍停留在个股页和当前股票。
- 页面无供应商品牌和内部错误原文。

### 10.4 快照兼容测试

- 新协议部署后，旧 `research_contract_version` 快照不能作为当日结论。
- 全量机会和主题筛选不能同时出现两个不同阶段名称。
- 旧快照可以作为历史复盘，但动作、评分和执行条件必须清空。

## 11. 非目标

- 不接真实交易和自动下单。
- 不安装未经验证的社区技能到生产主链。
- 不用大模型自由生成不可复现的核心评分。
- 不在本轮建设完整 DCF、组合优化器或机器学习预测系统。
- 不因为增加外部能力而移除本地事实和降级链路。
