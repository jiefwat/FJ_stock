# 问财四大工作台集成设计

## 目标

把已经在个股页验证可用的问财研究能力扩展到“每日大盘、我的持仓、个股分析、热点机会”四个核心工作台。每个页面只提供与当前任务相关的快捷核查，不增加通用聊天历史，不让外部结果改变 StockTs 本地风险闸门、评分、仓位或候选排序。

## 真实能力基线

现有 10 个个股研究技能继续保留：财务、机构预期、行业、事件、经营、股东治理、行情资金、公告、研报、新闻。

2026-07-14 使用生产服务器 Key 验证以下 4 个官方技能可通过现有 `/v1/query2data` 契约调用：

| 技能 | 实测结果 | 目标工作台 |
|---|---:|---|
| `hithink-zhishu-query` | 成功，3 条 | 每日大盘 |
| `hithink-macro-query` | 成功，1 条 | 每日大盘 |
| `hithink-sector-selector` | 成功，1 条 | 每日大盘、热点机会 |
| `hithink-astock-selector` | 成功，10 条 | 热点机会 |

四次响应均未回显 `IWENCAI_API_KEY`。

## 方案比较

### A. 共享研究坞与模块策略（采用）

四个工作台复用一个服务端 endpoint、一个安全客户端、一个证据卡渲染器；每个模块拥有独立的技能白名单、上下文构造器和快捷问题。优点是交互一致、安全边界集中、不会复制四套 JavaScript；代价是服务端必须严格校验模块上下文。

### B. 四套独立接口与组件

每个页面可完全定制，但会重复鉴权、限流、脱敏、错误处理和移动端样式，后续技能契约变化时容易出现页面间行为漂移。

### C. 页面加载时自动批量调用

首屏可直接出现外部数据，但会延长页面加载、消耗额度、放大外部故障，并可能在用户未选择目标时发送过多上下文。与当前“本地结论优先、外部证据按需核查”的方向冲突。

## 四模块能力映射

| 工作台 | 快捷核查 | 默认技能 | 发送的上下文 | 明确不发送 |
|---|---|---|---|---|
| 每日大盘 | 指数结构、宏观变量、主线板块、风险新闻 | 指数、宏观、行业/板块、新闻 | 问题、市场日期 | 持仓、成本、账号 |
| 我的持仓 | 业绩风险、公告核查、机构下修、资金异动 | 财务/事件、公告、机构、行情 | 单只持仓代码、名称、问题 | 股数、成本、权重、组合列表 |
| 个股分析 | 财务质量、机构预期、事件风险、行业位置及自由追问 | 现有 10 个个股技能 | 当前股票代码、名称、问题、本地日期 | 持仓、成本、Cookie |
| 热点机会 | 板块持续性、A 股筛选、事件催化、风险排除 | 板块筛选、A 股筛选、事件、新闻/行业 | 选中的板块或候选代码、名称、问题 | 本地候选评分、仓位、账号 |

模块路由必须限制在各自白名单内。例如大盘问题不能路由到个股财务技能，机会页的“A 股筛选”结果不能自动进入本地候选排名。

## 交互设计

四个页面使用同一个“问财外部核查”研究坞：

```text
+----------------------------------------------------------------+
| 问财外部核查                         已连接 / 未配置 / 需登录   |
| [模块快捷问题] [模块快捷问题] [模块快捷问题] [模块快捷问题]    |
| [上下文选择，仅持仓与机会需要]                                  |
| [输入一个需要外部数据验证的问题____________________] [核查]     |
+----------------------------------------------------------------+
| 技能 · 摘要 · 前 5 条事实 · 未知与限制 · 来源 · trace 尾号     |
+----------------------------------------------------------------+
```

- 默认只显示一行状态、快捷问题和输入区；有结果后在原位展开证据卡。
- 不显示头像、欢迎语、历史对话、背景故事和方法旁白。
- 本地结论永远排在研究坞之前，外部结果使用“核查”而不是“决策”措辞。
- 持仓页必须先选择一只持仓；选择器仅含名称和代码。
- 机会页可选择板块或候选。A 股筛选无须股票代码，但仍受机会总闸门约束。
- 桌面端单行操作，390px 下按“状态 -> 快捷问题 -> 上下文 -> 输入 -> 按钮 -> 结果”单列排列。
- 支持键盘提交、加载时禁用控件、可见焦点和 `prefers-reduced-motion`。

## 页面位置

- 每日大盘：本地核心判断和五步风险轨道之后、三情景与详细证据之前。
- 我的持仓：组合风控结论和处置队列之后、持仓证据折叠区之前。
- 个股分析：维持现有关键证据之后、风险反证之前的位置。
- 热点机会：机会总闸门和证据漏斗之后、候选与风险排除明细之前。

这种顺序保证用户先看到 StockTs 的确定性结论，再决定是否消耗外部额度补证。

## 服务端契约

继续使用 `POST /api/iwencai/research`，保持现有个股请求兼容。新增字段：

```json
{
  "module": "market | portfolio | stock | opportunity",
  "question": "不超过 200 字",
  "context": {
    "code": "可选，最多 32 字",
    "name": "可选，最多 64 字",
    "sector": "可选，最多 64 字",
    "local_as_of": "可选，最多 32 字"
  }
}
```

- 缺少 `module` 时按 `stock` 处理，兼容当前页面与测试。
- 服务端只读取白名单字段并清除控制字符，不接受浏览器提供技能 ID、endpoint、Base URL 或请求头。
- `portfolio` 必须有代码和名称；`opportunity` 至少有板块或候选；`market` 不接受股票或持仓上下文。
- 服务端根据 `module + question` 选择技能并拼装查询，浏览器不能绕过模块技能白名单。
- 响应延续 `summary / facts / relationship / unknowns / source / boundary`，新增 `module` 和 `context_label` 供页面审计。

## 代码边界

- `src/stock_ts/iwencai.py`：扩展 4 个技能、模块白名单、路由和查询上下文；继续负责 OpenAPI、脱敏、大小限制和证据压缩。
- `src/stock_ts/webapp/research_console.py`：新增共享研究坞 HTML 渲染器和四套快捷问题配置，不读取 Key、不发网络请求。
- `src/stock_ts/webapp/market_workspace.py`、`portfolio_workspace.py`、`opportunity_workspace.py`、`stock_workspace.py`：只决定研究坞插入位置和可选上下文。
- `src/stock_ts/webapp/shell.py`：把当前个股专用提交逻辑泛化为模块 payload，继续使用 `textContent` 安全渲染。
- `src/stock_ts/webapp/styles.py`：共享紧凑研究坞样式，仅为上下文选择器补充响应式规则。
- `src/stock_ts/web.py`：验证模块 payload、鉴权、按用户限流，并调用共享研究服务。

## 数据流

```text
本地工作台结论
  -> 用户选择模块快捷问题和最小上下文
  -> 同源登录态 POST
  -> 模块字段白名单与技能白名单
  -> 问财官方技能
  -> 最多 5 条结构化事实
  -> 外部核查证据卡
  -> 用户回到本地闸门、风险或个股档案复核
```

问财结果不写入本地快照、持仓账本、候选评分或交易计划。

## 安全与失败边界

- Key 继续只存在服务器环境，配置摘要只返回 `configured` / `missing`。
- 公网只允许登录用户调用；所有模块共享按用户限流，避免切页绕过额度限制。
- 问题 200 字、请求体 16 KiB、外部超时 12 秒、响应上限 1 MiB、最多展示 5 条事实。
- 持仓模块在前端、请求解析测试和服务端查询构造三个层面禁止 `shares`、`cost_price`、`weight`、用户名和 Cookie。
- 外部失败时只收起研究结果并显示可操作错误，本地四大工作台继续可用。
- stale/blocked 状态不因问财返回结果而解除；机会候选不因外部筛选结果自动升级为“可验证”。

## 验收标准

- 四个工作台都显示同一视觉语言的紧凑研究坞，且插入位置符合本设计。
- 14 个官方技能均有路由测试；新增 4 个技能通过真实服务器调用验收。
- 每个模块的 4 个快捷问题都只能路由到本模块白名单。
- 持仓请求与查询文本中不出现股数、成本、仓位、组合列表或用户信息。
- 旧个股 payload 继续可用；错误、限流、无 Key、未登录和空数据保持安全降级。
- JavaScript 不使用 `innerHTML` 渲染外部字段；桌面和 390px 无横向溢出。
- 专项测试、lint 和页面烟测通过；全量测试如仍有既有基线失败必须单独列明。
- 本地、GitHub、服务器 `main` 一致，公网健康页、登录保护和四模块真实查询均完成验收。
