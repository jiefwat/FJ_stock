# 四大原生研究工作台设计

## 背景

当前每日大盘、我的持仓、个股分析和热点机会仍以本地规则模型生成主结论，再把问财作为一个可选核查入口。这个结构有两个问题：页面同时存在“本地结论”和“外部核查”，用户需要理解两套口径；旧分析链仍要在首屏同步运行，页面慢且结论密度高。

新版本直接把问财数据技能作为四个模块的唯一研究引擎。供应商只存在于服务端适配层，产品界面只展示 StockTS 自己的研究结果模型。

## 目标

- 四个模块的核心结论全部来自服务端技能编排，不再消费旧的大盘、组合、个股和机会规则结论。
- 页面不出现“问财”“iWencai”“同花顺”“Skill”“外部证据”等供应商或实现术语。
- 首屏只回答：当前判断、现在怎么做、最大风险、三条关键发现。
- 页面壳立即可用，研究在进入模块后按需生成；单个能力失败不阻塞整页。
- 保留登录、限流、超时、响应大小、凭证隔离和持仓隐私边界。

## 方案比较

### 方案 A：首屏同步调用四个模块

实现直观，但会把多个外部请求串到页面首屏，慢、贵且容易因单点失败拖垮整页，不采用。

### 方案 B：原生研究工作台按需生成

页面先渲染四个轻量工作台。首次进入或点击“重新分析”时，调用模块 endpoint；服务端并行执行固定能力包，归一化为同一产品协议，并短时缓存。采用。

### 方案 C：保留本地结论作为兜底

兼容性最好，但继续形成双口径，也违背本分支“内部全部换成技能”的目标，不采用。失败时直接展示“证据不足”和重试动作，不回退旧结论。

## 服务端架构

```text
浏览器工作台
  -> POST /api/research/workspace
  -> ResearchWorkspaceService
  -> 模块固定能力包（并行、限时、可部分成功）
  -> 供应商响应归一化
  -> ResearchWorkspaceResult
  -> 短时内存缓存
```

新增 `src/stock_ts/research_engine.py`：

- 定义四个模块及固定能力包。
- 为市场、持仓、个股和机会构造受控查询。
- 并行调用已有安全客户端，不让浏览器指定能力 id、版本或网关。
- 把动态表格压缩为 `finding`、`detail` 和 `missing_section`。
- 仅根据明确返回事实和风险词生成条件化结论；证据不足时不伪造投资建议。
- 缓存键只包含模块和去隐私后的代码、名称、板块；缓存不包含股数、成本、权重或账号信息。

模块能力包：

| 模块 | 默认能力 | 页面目标 |
| --- | --- | --- |
| 市场判断 | 指数、宏观、板块筛选、新闻 | 市场状态、风险偏好、主线和风险事件 |
| 组合诊断 | 事件、公告、机构预期、行情资金 | 逐只持仓风险，不发送数量、成本或权重 |
| 公司研究 | 财务、经营、机构预期、事件 | 公司质量、预期变化、催化和主要风险 |
| 机会扫描 | 板块筛选、A 股筛选、事件、新闻 | 方向、候选、催化和风险排除 |

持仓模块最多分析前三只持仓。每只股票独立请求，只传代码和名称；不把完整组合、股数、成本、权重、备注、账号或 Cookie 发送给上游。

## 产品响应协议

`POST /api/research/workspace` 请求：

```json
{
  "module": "stock",
  "context": {"code": "600519", "name": "贵州茅台"},
  "refresh": false
}
```

响应只包含产品字段：

```json
{
  "ok": true,
  "status": "complete",
  "module": "stock",
  "generated_at": "2026-07-14T16:00:00+08:00",
  "verdict": "核心经营与预期证据已返回，当前进入条件复核",
  "action": "先核对三条关键变化，再决定是否进入价格验证",
  "primary_risk": "事件风险仍需持续复核",
  "findings": [],
  "details": [],
  "missing_sections": []
}
```

响应禁止包含供应商名称、能力 id、网关 URL、完整 trace id、API key 或内部异常堆栈。

## 交互设计

页面主题是“投研简报纸带”：深墨蓝、冷灰纸面、确认绿和风险红。信息像一张盘中判断单，而不是聊天机器人。

每个模块统一为：

```text
模块名                      最近生成时间
────────────────────────────────────
当前判断
一条主结论

现在怎么做                  最大风险

关键发现（最多三条）

[查看完整依据]  [重新分析]
```

- 首次进入自动生成一次；缓存命中时立即显示。
- 加载时使用一条横向“判断带”，不使用大面积骨架屏。
- 失败时只显示失败原因和“重新分析”，不展示旧规则结论。
- “继续追问”放在完整依据抽屉底部，使用原生模块语言；不呈现供应商名称、聊天头像或技能列表。
- 个股页保留代码/名称切换；持仓页保留持仓选择，但界面不展示数量、成本和权重给研究 endpoint。

## 视觉令牌

- `ink`：`#10263A`，主文字和判断带。
- `paper`：`#F3F6F7`，工作台底色。
- `surface`：`#FFFFFF`，结论纸面。
- `signal`：`#0E6A5C`，证据就绪和可继续。
- `risk`：`#B64A3C`，风险和失败。
- `line`：`#CBD5D9`，结构分隔。
- 标题沿用当前中文无衬线体系；数字与时间使用等宽字体，避免引入公网字体依赖。

唯一显著视觉元素是“判断带”：一条贯穿结论卡顶部的细长状态条，同时编码生成状态、证据完整度和新鲜度。其余卡片保持安静、低圆角和低对比，不增加装饰性渐变。

## 安全与可靠性

- 公网必须登录；本地匿名能力仍需显式开关。
- 按用户或 IP 限流；请求体最大 16 KiB。
- 单能力 12 秒超时，单次响应最大 1 MiB；模块请求可部分成功。
- API key 只从服务端环境读取，不进入 HTML、JSON、日志、测试或提交。
- 仅使用官方 HTTPS 网关；浏览器不能覆盖网关、能力 id 或版本。
- 不自动交易，不输出确定收益承诺；证据不足时明确暂停判断。

## 验收标准

- 新分支从干净 `main` 创建，原始脏工作区不受影响。
- 根页面四个核心模块不再运行或展示旧本地分析结论。
- 四模块结果均来自 `ResearchWorkspaceService` 的固定能力包。
- 正常页面 HTML、按钮、提示和 API 产品响应均不含供应商品牌与实现术语。
- 四页默认只展示一条结论、一条动作、一条最大风险和最多三项。
- 1280px 和 390px 无横向溢出；键盘焦点和 reduced motion 可用。
- 鉴权、限流、请求大小、超时、响应大小、凭证脱敏和持仓隐私测试通过。
