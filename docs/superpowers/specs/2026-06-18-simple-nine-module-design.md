# StockTS 简化九模块工作台设计

**背景**

当前页面承载了过多“研究指挥舱 / 决策仪表盘 / 投研框架 / 数据质量”等中间层概念，虽然信息完整，但不符合日常用户的操作心智。用户现在明确要求按九个直接可理解的模块组织软件：A股大盘、板块分析、涨停板、跌停板、智能选股、个股分析、持仓分析、每日复盘报告、消息渠道。

**目标**

把 StockTS Web 工作台重构为一个真正可操作、低学习成本的九模块软件壳。用户打开后不需要理解内部分析框架，只需要进入某个明确业务页面并完成查看、分析、维护、发送。

## 产品结构

页面不再以“工作区”组织，也不再暴露抽象模块名。

采用单层导航：

1. A股大盘
2. 板块分析
3. 涨停板
4. 跌停板
5. 智能选股
6. 个股分析
7. 持仓分析
8. 每日复盘报告
9. 消息渠道

顶部保留一个轻量总览条，但它不是独立模块，不进入侧边栏。

## 模块职责

### 1. A股大盘
- 展示交易日、市场热度、涨跌家数比、市场状态、北向资金、风险与明日观察。
- 只回答“今天市场能不能做、适合进攻还是防守”。

### 2. 板块分析
- 展示板块热度榜、主线板块、轮动与风险。
- 只回答“今天最强方向是谁、持续性如何”。

### 3. 涨停板
- 展示当日涨停家数。
- 从候选股票池原始截面中提取最强上涨股票，优先展示接近或达到涨停线的标的。
- 若真实涨停样本不足，页面应明确说明“当前数据源仅能近似识别强势涨幅股”。

### 4. 跌停板
- 展示当日跌停家数。
- 从候选股票池原始截面中提取最弱股票，帮助识别亏钱效应扩散方向。

### 5. 智能选股
- 由当前 `candidates` 模块收口而来。
- 名称改为“智能选股”，但继续保留风险提示、观察条件、排序依据。
- 不做确定性荐股口吻，只做研究排序与盘前准备。

### 6. 个股分析
- 保留当前最有价值的深度个股分析。
- 默认只展示：核心结论、多角度评分、技术结构、公告事件、操作计划。
- 降低视觉噪音，减少不必要的装饰块。

### 7. 持仓分析
- 保留当前持仓可增删改能力。
- 页面上方先给组合健康度、总盈亏、集中度、行业暴露。
- 页面中段给持仓列表与编辑区。
- 页面下方给具体处理建议。

### 8. 每日复盘报告
- 显示一份可复制的 Markdown 日报。
- 同时给出“今日一句话 / 三件要事 / 持仓风险 Top3”的摘要卡。
- 这是对外发送前的最后审阅页。

### 9. 消息渠道
- 聚焦消息发送，而不是系统体检。
- 配置项保留：邮件、企业微信、飞书、默认发送通道、默认发送样式。
- 保留“发送测试消息”和“发送今日复盘”两个动作。
- 数据源状态只作为辅助信息，不再主导页面。

## 交互原则

- 侧边栏只出现九个模块名称，不出现“workspace / command / framework / decision”之类内部概念。
- 每个模块只做一件事，页内不再做多层二级切换。
- 顶部工具栏只保留：股票代码/名称、数据源、更新视图。
- 文案统一专业、直接，不使用“研究指挥舱”“作战”“剧本”等过度包装话术。

## 视觉方向

采用“交易台 + 研究终端”的克制风格：
- 深海军蓝作为主色，黄铜色做重点强调。
- 标题清晰、模块边界明确、按钮数量少。
- 风险信息用标准标签呈现，不做花哨视觉比喻。
- 保持移动端可读，但优先服务桌面工作流。

## 实现边界

本轮重点是 Web 壳与模块组织重构，不新增重型外部依赖。

允许的新增：
- 涨停板 / 跌停板视图所需的轻量派生函数
- 新的模块映射与页面文案
- 测试更新

本轮不做：
- 新行情源接入
- 新数据库
- 完整券商账户联动
- 定时任务系统

## 测试要求

至少覆盖：
- 页面壳包含九个导航模块
- 模块内容包含“涨停板 / 跌停板 / 智能选股 / 每日复盘报告 / 消息渠道”
- 原有持仓交互不回退
- 消息渠道页表单和发送动作仍可用
- 全量 pytest 通过，lint 通过
