# Research + Portfolio Operating System Design

## Goal

将 StockTS 从“分析页面集合”重构为一个可长期使用的本地研究与组合一体化操作系统，围绕研究、组合、执行三条工作流组织能力，而不是围绕孤立模块堆叠信息。

## Product Thesis

系统每天只回答五个问题：

1. 今天市场是否适合承担风险。
2. 哪些方向值得继续研究。
3. 哪只股票值得进入交易准备。
4. 当前持仓哪些位置需要处理。
5. 今天结束后，明天具体做什么。

因此产品不再让用户自行在“大盘 / 板块 / 候选 / 个股 / 报告”之间跳转，而是提供三个工作区：

- 研究中枢：市场、板块、候选、个股、事件
- 组合中枢：持仓、成本、风险预算、调仓动作
- 执行中枢：今日动作、盘前清单、盘中验证、复盘归档

## User Model

默认面向单机个人用户，但边界按 2-5 人小团队可扩展设计：

- 当前所有数据仍落地在本地文件系统，不引入多用户账户体系。
- 页面语言从“演示 / 报告感”切换为“操作系统 / 工作台感”。
- 文件路径、CSV 术语、内部实现细节默认下沉，不暴露在主交互区。

## Information Architecture

### 1. 主入口：Operating System Shell

页面壳只保留高价值导航：

- 今日总控（Home）
- 研究中枢（Research）
- 组合中枢（Portfolio）
- 执行中枢（Execution）
- 归档与系统（Archive / System）

顶部工具条保留“标的 + 数据源”两个控制项，但以研究会话的语义出现。

### 2. 今日总控

首页承担“今天先干什么”的职责，内容包含：

- 市场风险灯
- 当前焦点标的
- 候选优先对象
- 组合最高风险项
- 今天行动序列

首页不是摘要堆叠，而是任务起点。

### 3. 研究中枢

研究中枢合并以下能力：

- 市场状态
- 板块主线
- 候选池排序
- 候选焦点
- 个股研究票据
- 公告 / 新闻 / 技术 / 交易计划

这里的目标是把“发现机会 → 验证机会 → 进入交易准备”串成一条线。

### 4. 组合中枢

组合中枢单独承担所有持仓动作：

- 持仓列表
- 添加 / 编辑 / 删除持仓
- 成本与盈亏
- 风险预算
- 调仓建议
- 单持仓未来处理剧本

它必须像一个真正能操作的持仓工作台，而不是报告段落。

### 5. 执行中枢

执行中枢把原本散落的结论收束为动作：

- 当前动作
- 买入 / 加仓触发
- 止损 / 减仓触发
- 止盈观察
- 盘中检查清单
- 收盘复盘关注点

### 6. 归档与系统

保留：

- 完整日报 / HTML 归档
- 数据质量
- 系统状态
- 多源能力矩阵

但它们不再抢占主工作流。

## Architecture Refactor

### 1. Web 代码结构

当前 `src/stock_ts/web.py` 同时承担：

- HTTP 入口
- 状态组装
- 样式
- 脚本
- 模块渲染
- 表单交互

这会继续阻碍产品化。重构目标是：

- `src/stock_ts/web.py`：只保留入口和请求处理
- `src/stock_ts/webapp/shell.py`：页面壳、CSS、脚本、导航
- `src/stock_ts/webapp/composition.py`：页面装配与上下文编排
- `src/stock_ts/webapp/workspaces.py`：各工作区渲染
- `src/stock_ts/webapp/forms.py`：表单与跳转 URL 生成
- `src/stock_ts/webapp/view_models.py`：轻量页面上下文模型

### 2. Page Composition Model

Web 页面应先组装一个“界面上下文”，再渲染，而不是在模板里即席推导：

- 当前研究标的
- 市场分析结果
- 板块分析结果
- 候选池与候选焦点
- 组合分析结果
- 数据质量结果
- 公告 / 技术 / 交易计划结果
- 顶层 workspace 切换状态

### 3. Navigation Contract

所有按钮必须属于以下三类之一：

- 工作区切换按钮：只改变 `#workspace-*`
- 视图上下文按钮：改变 query string，并显式带回目标 workspace
- 提交按钮：POST 后重定向回明确 workspace

禁止出现“能点但不知道去了哪”的弱行为。

## UX Rules

- 默认把用户留在当前工作流，不做无提示跳回首页。
- 删除动作必须二次确认。
- 编辑态必须明显提示“正在编辑哪只股票 / 哪个候选”。
- 候选池、持仓、个股之间的跨区跳转必须保留上下文。
- 页面文案统一成专业软件语气，去掉口语化、演示化表达。

## Testing Strategy

### 必须保持

- `make lint`
- `make test`

### 新增重点

- 页面壳渲染测试：工作区存在、默认 workspace 正确
- 表单跳转测试：候选 / 个股 / 持仓按钮带回正确 hash
- 持仓交互测试：添加、编辑、删除流程
- Web GET 冒烟：首页、研究中枢、组合中枢可返回 200

## Out of Scope

本轮不做：

- 多用户登录
- 云同步
- 实时推送
- 自动下单
- 真实券商接入
- SaaS 化后台

## Delivery Definition

交付完成的标志不是“页面更好看”，而是：

1. 用户可以自然完成“研究 → 组合 → 执行”闭环。
2. Web 入口不再像 HTML 拼装页，而像一个稳定的软件壳。
3. 交互按钮全部有效，且行为可预测。
4. 核心 Web 代码已拆出清晰边界，后续能继续演进。
5. 全量 lint / test / Web 冒烟通过。
