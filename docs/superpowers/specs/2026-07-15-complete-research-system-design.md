# 完整股票研究系统设计

## 目标

把当前四个可按需调用的研究工作台升级为一套可以长期使用的完整系统：用户能看大盘趋势、全部持仓、专业个股研究和每日机会股票；系统能自动运行、持久化结果、失败降级并部署到公网。

页面继续坚持精华优先，不恢复旧版长报告。完整性由全量证据、覆盖率、自动快照和可追溯时间保证，而不是把所有内容同时堆在首屏。

## 方案选择

### 方案 A：扩展现有研究引擎并增加每日快照（采用）

沿用供应商中性的研究结果协议，扩展能力包、覆盖范围和模块专属列表；新增原子 JSON 快照、每日任务和 systemd timer。实时研究与每日快照共用同一个引擎，不产生两套结论。

优点是边界清晰、可分步验证、能直接复用当前页面和公网部署。缺点是研究引擎需要增加持久化与模块专属展示逻辑。

### 方案 B：恢复旧本地分析并与新引擎拼接

旧系统有较多技术指标和本地日报能力，但与当前服务端研究能力的字段口径不同。拼接后同一模块会出现两套结论，页面复杂度和维护成本都会回升，因此不采用。

### 方案 C：独立数据库与微服务

把采集、调度、研究和 Web 拆成多个服务，长期扩展性最好，但当前单机规模不需要微服务治理。本阶段不采用。

## 总体架构

```text
公网 Web / 登录用户
  -> 研究工作台 API
     -> 新鲜每日快照（大盘 / 机会）
     -> 实时研究引擎（刷新 / 持仓 / 个股）
        -> 固定能力白名单
        -> 语义证据门禁
        -> 模块结果协议
     -> 原子 JSON 快照仓库

systemd timer
  -> scripts/run_daily_research.py
     -> 大盘研究
     -> 每日机会筛选
     -> latest + 日期归档 + status
```

外部能力仍只通过服务端适配器调用。浏览器不能传入能力 id、版本、网关或凭证。

## 四个模块

### 1. 每日大盘

大盘页必须回答：市场处于什么趋势、风险预算应收缩还是保持、当前主线是什么、哪些事件可能推翻判断。

数据结构：

- 上证指数、深证成指、创业板指。
- 最新点位、当日涨跌幅、成交额。
- 能返回时展示 5 日和 20 日涨跌、5 日和 20 日均线或等价趋势字段。
- 行业/概念主线前 5。
- 最新市场事件和宏观证据。
- 生成时间、证据日期、完整度和缺口。

展示结构：

- 首屏继续显示一个市场判断、一个动作和一个最大风险。
- 判断下方增加三个指数趋势卡，不隐藏在完整依据里。
- 三条关键发现保持主线、事件和宏观优先。
- 多周期字段拿不到时明确显示“趋势周期待补”，不能用当日涨跌冒充中期趋势。

### 2. 我的持仓

持仓页必须覆盖当前账号的全部持仓，而不是只分析前三只。对外请求只携带代码和名称，不携带股数、成本、权重、备注、用户 id 或 Cookie。

范围与性能：

- 单次最多分析 20 只持仓，当前 11 只必须全部覆盖。
- 每只至少获取事件风险、机构预期和价格成交三类证据。
- 请求并发上限为 8，单能力继续使用既有超时。
- 首页只前置风险最高的 3 只；完整持仓清单展示全部股票和覆盖状态。

排序规则：

1. 负面事件、业绩预亏、监管、质押、解禁优先。
2. 预期下修或缺失其次。
3. 价格/成交异常其次。
4. 无明确风险的股票排在后面，标记为“暂未发现高优先风险”，不写成安全。

### 3. 个股分析

个股页从 4 维扩展为 8 维：

1. 财务质量。
2. 经营结构。
3. 机构预期。
4. 事件风险。
5. 价格与成交。
6. 行业位置。
7. 公告核查。
8. 研报观点。

首屏优先级仍为最新事件、财务方向、机构预期。价格、行业、公告和研报进入可见的“八维覆盖”状态条与完整依据，不新增大段方法论旁白。

结果必须包含：

- 股票名称与代码。
- 一句话判断、下一步动作、最大风险。
- 8 维状态和覆盖率。
- 最多三条首屏发现。
- 全量事实、缺口、来源日期和生成时间。

### 4. 每日机会

机会页必须形成“今天可以继续研究哪些股票”的每日清单，不输出保证上涨或直接买入结论。

每日结果：

- 行业/概念方向前 5。
- 股票候选至少 5、最多 10；接口不足时按实际数量展示并说明缺口。
- 每只候选包含代码、名称、核心入选依据、至少一个风险/失效提示和可用的财务或成交事实。
- 明确生成日期与时间，区分今日快照、过期快照和实时刷新。
- 事件与新闻用于风险排除，不能把无目标的负面事件直接显示成候选理由。

页面结构：

- 首屏判断、动作和最大风险。
- 主线、候选首位、风险事件三条发现。
- 新增“今日机会清单”，默认展示最多 10 只，不要求展开完整依据。
- 每只候选提供进入个股分析的入口。

## 每日快照

新增 `stock_ts.research_snapshots`：

- `reports/research/market/latest.json`
- `reports/research/market/YYYY-MM-DD.json`
- `reports/research/opportunity/latest.json`
- `reports/research/opportunity/YYYY-MM-DD.json`
- `reports/research/daily.status.json`

写入必须使用临时文件加原子替换，避免任务中断留下半个 JSON。

快照读取规则：

- 页面首次加载大盘和机会时优先读取 18 小时内的成功快照，实现秒开。
- 用户点击“重新分析”时执行实时研究，并覆盖最新快照。
- 实时研究失败时允许返回最后一次成功快照，但必须标记 `stale=true` 和生成时间。
- 没有快照且实时失败时才显示暂停判断。
- 持仓和个股不写入共享每日快照，避免用户上下文混入公共结果。

## 自动任务

新增 `scripts/run_daily_research.py`，只运行大盘和机会两个全局模块，输出快照与状态。命令支持：

```bash
PYTHONPATH=src python scripts/run_daily_research.py \
  --output-dir reports/research \
  --refresh
```

新增独立 systemd 单元：

- `stock-ts-daily-research.service`
- `stock-ts-daily-research.timer`

定时点使用 Asia/Shanghai：07:20、12:10、18:30。`Persistent=true`，服务器错过执行点后恢复时补跑。独立于旧 TDX 日报流水线，任一任务失败不阻塞另一个任务。

## 产品协议

现有 `verdict / action / primary_risk / findings / details / missing_sections` 保持兼容，新增：

- `subject_count`：本次覆盖对象数。
- `coverage`：完成维度数和总维度数。
- `delivery`：`live`、`snapshot` 或 `stale_snapshot`。
- `as_of`：结果数据时间。
- `module_items`：模块专属列表；分别承载指数、持仓、八维状态或机会候选。

`module_items` 只使用产品字段，不包含供应商品牌、能力 id、trace、网关或授权信息。

## 页面交互

- 保留当前研究行动坞和移动底部导航。
- 大盘显示三个指数趋势卡。
- 持仓显示“已分析 11/11”及全量持仓处理清单。
- 个股显示八维覆盖条。
- 机会显示今日候选清单和快照时间。
- 首屏仍不超过判断、动作、风险和三条重点；模块专属清单位于关键发现之后。
- 390px 无横向滚动，候选和持仓卡单列显示。

## 错误与新鲜度

- 单能力失败：模块降级为 `partial`，保留其他证据。
- 全部实时失败：优先使用最后成功快照；没有快照才暂停判断。
- 快照超过 18 小时：标记过期，不伪装为今日结果。
- 候选缺少代码或名称：不进入今日机会清单。
- 新闻和公告超过 30 天：不进入首屏，只能留在完整依据并标记日期。
- 持仓超过 20 只：分析前 20 只并明确覆盖边界。

## 安全边界

- 公网继续要求登录，注册入口保留。
- 不开放匿名外部研究请求。
- 持仓仅传代码和名称。
- 快照不保存账号、持仓、Cookie 或凭证。
- 所有动态文本继续使用 `textContent`，不使用 `innerHTML`。
- 保持用户/IP 限流、请求大小、响应大小和能力超时限制。

## 验收标准

1. 大盘页展示三个指数趋势项、主线、事件、生成时间和缺口。
2. 当前 11 只持仓全部进入研究结果，页面显示覆盖数和全量清单。
3. 个股页显示 8 个研究维度，完整结果至少 6 维可用时才允许 `complete`。
4. 机会页展示 5 至 10 只带代码、名称、依据和风险的候选。
5. 每日脚本生成 latest、日期归档和 status，重复运行安全。
6. systemd timer 在服务器启用，三个定时点可见，手动运行成功。
7. 快照新鲜时页面优先使用快照；刷新走实时；实时失败可降级到过期快照。
8. 1280px 和 390px 均无横向溢出，移动行动坞不遮挡内容。
9. 产品 HTML/JSON 不出现供应商品牌、内部能力、trace、网关或密钥。
10. 本地 `main`、GitHub `main`、服务器 `main` 一致，公网登录后四模块真实请求通过。
