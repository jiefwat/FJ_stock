# 大盘事实台与热门机会预测闭环设计

## 目标

把“每日大盘”和“热门机会”拆成两个职责明确、不可混用的研究工作区：

```text
每日大盘 = 截至当前已经发生并可复核的市场事实
热门机会 = 面向未来 1/3/5/10 个交易日的条件化研究预测
```

用户点击热门主题后，可以看到主题判断、支持证据、主要反证、持续性和主题股票；点击股票后进入个股分析。持续性观察名单最多展示 Top10，但只展示通过硬闸门的股票，禁止用低质量候选凑数。

“可进入投资候选”表示数据、流动性、风险和研究证据通过候选门槛，不表示保证上涨，也不构成无条件买入指令。

## 已确认现状

- 生产服务和现有三个定时任务正常运行，当前部署基线为 `e572aae`。
- 线上快照覆盖全市场截面，但最近流水线因部分持仓 K 线落后市场交易日而失败。
- 当前已有 3/5/10/20 日收益、上涨分布、量能、回撤、均线、过热和单日脉冲判断，可作为前瞻候选的价格结构基础。
- 当前大盘仍包含“持续性观察名单”，与大盘只展示事实的新职责冲突。
- 当前机会页已有候选与个股入口，但没有完整主题下钻、预测历史台账和自动结果反馈。
- 外部公开异动源可能忽略历史日期参数并重复返回同一批数据，因此所有外部证据必须校验发布时间、新鲜度和目标对象，不得直接接管排名。

## 方案选择

### 方案 A：事实台 + 预测台 + 不可变预测台账（采用）

大盘使用确定性市场快照，机会使用多周期模型和公开证据，预测写入不可变台账，并在未来交易日自动回评。

优点：职责清晰、结论可复算、预测效果可审计，外部信息失效时仍可安全降级。缺点：需要增加主题路由、预测存储和回评逻辑。

### 方案 B：只调整页面分区

上线快，但无法知道历史候选是否有效，也不能阻止同一套事实被两个模块重复解释，不采用。

### 方案 C：实时全网情报直接驱动推荐

信息丰富，但来源时间戳、稳定性和可复现性不足，会放大热点追涨和幸存者偏差，不采用。

## 产品职责

### 每日大盘：事实控制台

每日大盘只回答“市场刚刚发生了什么”，固定展示：

1. 数据交易日、生成时间、核心数据覆盖和新鲜度。
2. 上证、深证、创业板等指数的收盘、涨跌和成交事实。
3. 上涨、下跌、平盘、涨停、跌停家数和市场宽度。
4. 成交额、量能变化、强弱分布和市场温度事实。
5. 当日主题表现、参与股票数量和内部扩散程度。
6. 当日异常上涨、异常下跌和风险事件。
7. 数据缺口与受影响模块。

大盘不得展示：

- 预期收益、未来上涨判断或“可投资”标签。
- 持续性候选、推荐股票或买卖动作。
- 仅由单日涨幅推导出的明日结论。
- 未经时间校验的外部热度或社区观点。

大盘中的主题可以点击进入热门机会的同名主题研究，但大盘卡片本身只陈述当日事实。

### 热门机会：前瞻研究台

热门机会只回答“哪些方向值得在未来几个交易日继续验证”，固定展示：

1. 机会总闸门：市场和数据是否允许形成前瞻候选。
2. 热门主题列表：主题假设、未来观察窗口、支持证据、最大反证、确认条件和失效条件。
3. 主题下钻：主题股票、阶段、数据日期、价格结构、证据覆盖和个股分析入口。
4. 持续性观察 Top10：跨主题统一排序后的前瞻候选。
5. 风险排除：过热、脉冲、陈旧、低流动性、重大负面或证据不足的股票及原因。
6. 历史反馈：过去预测的命中、失效、超额收益和误判原因。

前瞻阶段固定为：

| 阶段 | 含义 | 默认展示位置 |
| --- | --- | --- |
| `可进入投资候选` | 通过全部硬闸门，可进入后续个人决策 | Top10 |
| `等待确认` | 结构或证据接近门槛，等待可量化确认 | 主题下钻 |
| `只观察` | 有研究价值但风险、估值或波动不适合当前介入 | 观察区 |
| `排除` | 数据、流动性、过热或事件风险触发硬闸门 | 风险排除 |

页面不使用“必涨”“稳赚”“直接买入”等确定性表达。

## 主题下钻与路由

保持现有单页应用和登录逻辑，不引入新服务或独立页面。

### 主题入口

主题链接采用：

```text
/?theme=<URL编码主题>&provider=<provider>#opportunity
```

大盘主题和机会主题使用同一参数。页面加载后：

- 自动激活 `opportunity` 工作区。
- 在顶部显示当前主题研究标题和返回全部主题入口。
- 只筛选主题名称或标准化别名匹配的候选。
- 保留主题级支持证据、反证、确认和失效条件。

### 个股入口

主题股票链接采用：

```text
/?code=<股票代码>&source_theme=<URL编码主题>&candidate_source=opportunity#stock
```

个股页显示“来自热门机会 / 主题名称”的来源上下文，但重新使用个股模块自己的完整数据闸门和八维结论；主题页结论不能绕过个股页风险判断。

### 安全约束

- 参数只参与标准化筛选和安全文本渲染，不拼接 HTML。
- 未知主题返回空结果和可读提示，不回退到全市场候选。
- 主题名称、股票名称和外部摘要统一转义。
- 不向页面暴露第三方平台名称、Logo、能力 ID、trace、网关字段或凭证。

## Top10 投资候选门槛

Top10 从至少 50 只、目标 100 只已完成深度补强的候选中产生。候选必须先通过硬闸门，再按前瞻研究分排序。

### 硬闸门

候选必须同时满足：

- 最新可靠日线与市场交易日一致。
- 至少 20 根真实日线，价格可靠性通过。
- 不属于 ST、退市整理或明确不可交易状态。
- 近阶段成交额和换手满足可执行流动性门槛；数据缺失时不得进入最高阶段。
- 未触发涨幅过热、均线偏离过大、异常波动或单日脉冲硬扣分。
- 没有未解除的重大负面公告、监管、减持、诉讼或财务风险闸门。
- 至少具备价格结构、主题归属和一项公司/事件证据。

### 排序维度

总分仍为研究排序，不映射为上涨概率：

| 维度 | 权重 | 口径 |
| --- | ---: | --- |
| 多周期趋势与上涨分布 | 25 | 3/5/10/20 日结构，不让最新一日主导 |
| 相对强度 | 15 | 相对指数和主题的 5/10/20 日表现 |
| 回撤、波动和过热风险 | 15 | 最大不利波动优先扣分 |
| 量能和流动性 | 15 | 多日成交质量，不用单日放量替代持续性 |
| 主题扩散和持续性 | 10 | 主题内有效股票数量、跨日稳定度和内部一致性 |
| 基本面与估值 | 10 | 真实数据存在才计分，缺失不补中性分 |
| 事件催化与反证 | 10 | 公开、具日期、可核查，反证与催化同时呈现 |

评分相同时依次按超额收益稳定度、最大回撤、数据覆盖度和股票代码排序，保证结果可重复。

### 数量规则

- 最多展示 10 只。
- 只有通过硬闸门且阶段为 `可进入投资候选` 的股票进入 Top10。
- 通过股票少于 10 只时展示实际数量，并解释主要缺口。
- 不允许使用单日涨幅榜、旧 K 线或低质量候选补足 10 只。

## 公开信息证据

公开信息只作为证据增强层，不直接设置主题阶段或股票排名。

### 可使用信息

- 交易所公告、公司公告和监管信息。
- 财经新闻、产业政策和公司事件。
- 市场异动及其关联原因。
- 公司财务、估值、机构预期和行业数据。
- 社区讨论只作为待核查线索，不作为事实。

### 证据契约

每条外部证据至少包含：

```text
标题 / 摘要 / 目标股票或主题 / 发布时间 / 抓取时间 /
证据类型 / 支持或反证 / 新鲜度 / 可核查状态
```

以下情况不得进入首屏结论：

- 缺少发布时间或发布时间晚于抓取时间异常。
- 查询日期与返回内容日期不一致。
- 同一内容跨日期重复且无法证明是持续事件。
- 只有社区情绪，没有公告、行情或公司事实佐证。
- 与目标股票或主题的关联只能靠关键词猜测。

所有外部能力继续在后台运行，页面保持供应商中性。

## 预测台账与自动反馈

### 不可变预测记录

每天研究任务完成后，为每个进入机会页的主题和股票写入预测记录：

```text
prediction_id
created_at / baseline_trade_date / baseline_price
subject_type / subject_code / subject_name / theme
stage / score / confidence
horizons = 1,3,5,10
support / counter_evidence
confirmation / invalidation
market_benchmark / theme_benchmark
data_as_of / evidence_as_of
model_version / snapshot_fingerprint
```

历史记录只追加，不覆盖原始阶段、分数和理由。相同交易日、对象、模型版本和基准快照使用稳定键去重，避免定时任务重复写入。

建议使用仓库已有的数据目录落地 JSONL 或 SQLite；首版优先选择 SQLite，以支持幂等写入、按交易日回评和聚合查询。数据库不得包含账号、持仓、成本或用户备注。

### T+1/T+3/T+5/T+10 回评

现有每日研究任务在产生新预测前，先回评已到期记录：

- 绝对收益。
- 相对大盘和主题的超额收益。
- 最大有利波动 MFE。
- 最大不利波动 MAE。
- 确认条件是否触发及触发日期。
- 失效条件是否触发及触发日期。
- 结果：`命中`、`未命中`、`已失效`、`已过期`、`数据不足`。
- 原始理由是否得到后续事实支持。

缺少未来 K 线或基准数据时固定为 `数据不足`，不得按失败或成功计算。

### 聚合反馈

机会页展示：

- 最近 20 个有效预测的 1/3/5/10 日命中率。
- 平均超额收益、中位超额收益和平均 MAE。
- 按阶段、主题和置信度的校准结果。
- 最近主要误判原因：追高、主题退潮、催化落空、流动性不足、数据陈旧等。
- 当前样本量和统计口径。

样本量小于 20 时只展示样本事实，不输出“模型有效”结论。单次结果不自动修改权重；权重调整必须有版本号、最小样本量和离线回放测试。

### 用户反馈

登录用户可对预测标记：

- 有用 / 没用。
- 原因正确 / 原因错误。
- 已关注 / 已忽略。

用户反馈按账号隔离，只用于复核队列和产品体验统计，不直接提高候选分数，不暴露给其他账号。

## 数据流与模块边界

```text
市场快照
  -> 大盘事实聚合器
  -> 每日大盘视图

候选快照 + 多周期模型 + 公开证据
  -> 前瞻研究闸门与排序
  -> 主题研究 / Top10 / 风险排除
  -> 不可变预测台账
  -> T+1/T+3/T+5/T+10 回评
  -> 聚合反馈
```

边界要求：

- 大盘事实聚合器不得依赖预测台账。
- 预测排序复用 `continuation.py` 的多周期统计，但不在 Web 层重新计算规则。
- 主题筛选和路由只消费标准化主题字段。
- 回评服务只读取历史预测和后续行情，不读取未来日期之外的信息重写原始记录。
- 外部证据失败时保留本地候选，但降级阶段和置信度。
- 不新增 systemd 定时器，复用现有每日研究任务。

## 每日晨报输出

晨报是大盘事实台、持仓研究和热门机会预测台的行动摘要，不再从长篇 Markdown 中拼接互相冲突的旧标题。它优先读取结构化决策、前瞻候选和预测反馈；结构化产物缺失时才安全降级到 Markdown 摘要。

晨报保持“30 秒读完”，不复制 Top10 全表，固定为以下五段：

### 1. 最新市场事实

最多三行，只陈述上一有效交易日或最新盘中快照已经发生的事实：

- 指数、涨跌宽度和成交额的核心变化。
- 涨停、跌停和极端涨跌分布。
- 当日最强事实主题及参与股票数量。

标题必须携带交易日；数据过期时写明“数据截至 YYYY-MM-DD”，不得使用“今天市场”或“当前行情”。这一段不出现预测、候选或仓位建议。

### 2. 先处理持仓

最多三只，按红灯、黄灯、绿灯和用户组合风险排序。每只固定包含：

```text
股票｜当前判断｜今天动作｜触发条件｜失效或离场线
```

持仓动作继续使用用户隔离的组合数据，但不得把成本、股数或个人备注带入共享日报、预测台账或公开链接。没有必须处理的持仓时明确写“暂无优先处置项”，不使用候选股填充。

### 3. 今日前瞻机会

最多三只，从 Top10 中按阶段、跨周期稳定度和执行可读性筛选，不能从未过闸候选补位。每只固定包含：

```text
股票｜主题｜阶段｜观察窗口｜支持证据｜最大反证｜确认条件
```

- 只允许 `可进入投资候选` 和高质量 `等待确认` 进入晨报。
- `等待确认` 必须显式标记，不能写成可买结论。
- 只有 1–2 只通过时只展示实际数量。
- 每只股票附带回到个股分析的链接，主题附带回到主题研究的链接。
- 晨报不展示外部平台名称、Logo、能力 ID 或来源品牌。

### 4. 预测反馈

只展示一条可校准的历史结果：

```text
近 N 个有效预测｜3 日命中率｜3 日平均超额｜平均 MAE｜主要误判原因
```

- 首版以 3 日窗口为晨报主指标，1/5/10 日完整结果留在机会页。
- 有效样本少于 20 时固定显示“样本积累中”，只报告样本量和已发生结果，不评价模型有效性。
- 没有到期预测时显示“暂无到期样本”，不得隐藏为零命中率。

### 5. 数据风险

最多两行，只保留会改变当天判断的阻断项：K 线过期、持仓数据缺失、候选深度覆盖不足、预测回评缺失或关键任务失败。内部命令、异常堆栈、服务器路径和供应商名称不得进入邮件。

晨报末尾只保留一条纪律：`先处理持仓风险，再验证机会；没有确认，不扩大风险。` 同时提供大盘、持仓和热门机会三个站内入口，并保留一条简洁的“不构成投资建议”声明；不附带长报告路径、运维状态或重复免责声明。

### 晨报数据优先级

```text
结构化大盘事实
  + 账号隔离的持仓决策
  + 前瞻 Top10 / 主题路由
  + 预测反馈聚合
  -> 30 秒晨报
  -> 邮件 dry-run
  -> 现有发送渠道
```

晨报生成失败不阻断每日研究任务；发送前继续支持 dry-run。若大盘事实日期、Top10 数据日期和持仓行情日期不一致，晨报必须在数据风险段提示，并把机会阶段最高降为 `等待确认`。

## 视觉与交互

保持当前纸白、墨蓝、克制绿红的研究终端语言，不改变登录、导航、持仓和个股的现有页面逻辑。

### 每日大盘

视觉重点是“事实时间轴”：每项关键事实都带交易日或生成时间。主题使用可点击的事实条，展示当日强度、参与数和扩散度，不显示预测标签。

### 热门机会

视觉重点是“研究漏斗”：

```text
机会总闸门 -> 热门主题 -> 主题股票 -> Top10 -> 历史反馈
```

主题卡明确可点击，选中主题后保持高亮并显示股票列表。Top10 使用紧凑列表，不使用大面积重复卡片；每行必须同时显示入选理由、最大反证、确认和失效条件。移动端保持单列，主题带可横向滚动，页面本身不得横向溢出。

## 错误处理与降级

- 大盘核心事实过期：显示数据日期和缺口，隐藏所有“截至今日”表达。
- 候选 K 线过期：从 Top10 排除，并进入风险排除区。
- 主题数据缺失：股票可保留在全部候选，但不能进入主题 Top10。
- 外部证据失败：使用本地事实，阶段最高为 `等待确认`，并显示证据缺口。
- 台账写入失败：不阻断大盘和机会页，任务状态记录失败并在数据质量页提示。
- 回评数据不足：保持待评估，不计入命中率分母。
- 未知主题参数：显示空状态，不回退到未筛选列表。

## 测试策略

按 TDD 顺序实施：

1. 大盘只包含事实、没有候选和未来语言的契约测试。
2. 机会页主题链接、主题筛选、股票链接和来源上下文测试。
3. Top10 硬闸门、排序稳定性、少于 10 只不补位测试。
4. 外部证据时间戳、重复日期和供应商中性测试。
5. 预测台账幂等写入、不可变字段和模型版本测试。
6. T+1/T+3/T+5/T+10 收益、超额收益、MFE、MAE、确认和失效测试。
7. 用户反馈账号隔离和不改变评分测试。
8. 晨报五段顺序、条数上限、日期一致性、链接和无品牌测试。
9. 晨报少于三只不补位、反馈样本不足和 dry-run 降级测试。
10. 桌面、390px 移动端、路由和登录回归测试。
11. 每日研究任务集成测试，确认不新增定时器且失败安全。
12. 公网部署后健康、登录边界、主题下钻、个股入口、晨报 dry-run 和供应商中性烟测。

## 验收标准

1. 每日大盘首屏只展示已发生事实、数据时间和数据缺口，不出现预测候选或买卖语言。
2. 热门机会明确是未来研究，候选包含 1/3/5/10 日观察窗口、支持、反证、确认和失效。
3. 点击大盘或机会主题可进入同名主题研究，显示对应股票。
4. 点击主题股票可进入个股分析，并保留来源主题上下文。
5. 持续性名单最多 Top10；少于 10 只时不补位并解释缺口。
6. Top10 所有股票通过行情新鲜度、真实 K 线、流动性、过热、风险事件和证据门槛。
7. 历史预测按 T+1/T+3/T+5/T+10 自动回评，原始预测不可覆盖。
8. 机会页展示样本量、命中、超额收益、MAE 和近期误判原因，数据不足时不夸大有效性。
9. 用户反馈按账号隔离且不直接改变股票排序。
10. 页面、网络响应和公开产物不出现第三方 Logo、供应商名称、能力 ID、trace 或凭证。
11. 现有登录、持仓编辑、个股分析、日报、通知和三个定时任务保持可用。
12. 晨报按市场事实、持仓、三只以内前瞻机会、预测反馈和数据风险输出，30 秒可读且不混淆事实与预测。
13. 晨报候选不足三只时不补位，样本不足时不夸大预测有效性，并保留站内主题和个股入口。
14. 专项测试、核心回归、Lint、本地浏览器、晨报 dry-run 和公网烟测通过后才能宣称部署完成。

## 非目标

- 不连接券商、不自动下单、不管理真实资金。
- 不承诺收益，不把研究分数解释成上涨概率。
- 不通过社区热度直接推荐股票。
- 不为了展示 Top10 而放宽数据和风险闸门。
- 不重做整站导航、登录、持仓或个股页面逻辑。
- 不新增公开第三方品牌入口或定时服务。
