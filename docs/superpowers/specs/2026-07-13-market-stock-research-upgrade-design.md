# StockTS 大盘与个股专业研究升级设计

日期：2026-07-13

状态：已授权设计方向，待实施计划

范围：仅升级“每日大盘”和“个股分析”两个工作区，以及它们直接依赖的研究模型、数据契约与测试；不改持仓、热点机会、账户和通知产品逻辑。

## 1. 背景与判断

StockTS 已有行情、市场宽度、板块、技术指标、资金、新闻公告、基础财务、组合成本、数据质量和 TradingAgents 风格决策链。当前问题不是功能数量不足，而是研究层级与界面叙事没有形成稳定闭环：

- 大盘页面展示了涨跌统计、强弱板块和买卖指导，但尚未把市场阶段、风格、流动性、赚钱效应、风险预算和失效条件组织成统一判断。
- 个股页面具备六维证据和多角色分析，但大量内容仍由规则分数、通用阈值和固定价格比例驱动，缺少公司质量、盈利趋势、估值锚、预期差和情景推演。
- `src/stock_ts/web.py` 同时承担编排、研究文案和 HTML 渲染，模块边界过大，增加继续深化研究和验证的成本。
- 当前测试大量验证页面关键词是否存在，能防止结构丢失，但不足以证明市场判断和个股结论在数据变化时保持一致、可解释和可降级。

本次升级把产品定位从“行情与规则看板”推进为“个人研究终端”。页面不追求展示更多字段，而追求让每个结论都能回答：依据是什么、反证是什么、数据是否足够、何时失效。

## 2. 设计原则

1. 结论先于指标，但结论必须能回溯到证据。
2. 数据完整性是硬闸门，不用缺失字段推导强结论。
3. 分数只用于排序和压缩信息，不能代替研究论点。
4. 大盘决定风险预算，个股决定是否值得承担这部分风险。
5. 研究与交易计划分离：研究回答“值不值得继续跟踪”，交易计划回答“满足什么条件才行动”。
6. 所有预测使用情景、概率、触发和失效表达，不输出确定收益承诺。
7. 保留当前中文、紧凑、操作台式界面，不引入通用金融仪表盘模板。

## 3. 方案选择

### 方案 A：只重排当前页面

优点是改动小、上线快。缺点是现有研究模型仍以规则分和固定阈值为主，只能改善观感，无法解决“浅”和“不专业”。不采用。

### 方案 B：只增强研究模型

优点是能补财务、估值和情景分析。缺点是现有页面会继续堆叠报告片段，用户难以快速理解主要矛盾。作为后端工作的一部分采用，但不单独实施。

### 方案 C：研究模型与决策界面协同升级

同时建立市场状态模型、个股投资备忘录、证据审计和专业界面。改动较大，但能形成从数据到判断再到失效条件的完整链路。本次采用此方案。

## 4. 每日大盘：市场状态机

### 4.1 页面必须回答的问题

首屏在 20 秒内回答：

1. 当前市场阶段是什么：进攻、轮动、震荡、防守或风险释放。
2. 今日风险预算是多少：总风险暴露区间、单票上限和禁止动作。
3. 赚钱效应在哪里：指数、风格、行业/主题和强度持续性。
4. 哪些条件会改变判断：升级触发、降级触发和关键风险事件。

### 4.2 市场状态模型

新增 `MarketRegimeAssessment`，由下列五个维度组成：

| 维度 | 主要输入 | 输出 |
| --- | --- | --- |
| 趋势 | 主要指数、均线位置、阶段高低点 | 趋势方向与一致性 |
| 宽度 | 上涨/下跌、涨停/跌停、分位分布 | 参与度与扩散程度 |
| 流动性 | 成交额、成交额变化、北向/可用资金代理 | 放量、缩量与风险偏好 |
| 风格 | 大小盘、成长/价值、行业/主题相对强弱 | 主导风格与切换信号 |
| 情绪 | 连板/炸板可用项、极端涨跌、风险新闻 | 情绪阶段与退潮风险 |

每个维度必须输出 `status`、`evidence`、`confidence`、`missing_fields`。最终市场阶段由规则组合生成，并附带最强支持证据、最强反证和失效条件。缺少流动性或风格数据时可以输出阶段判断，但置信度必须下降；行情日期过期时直接进入“数据暂停”。

### 4.3 页面结构

桌面端采用从判断到证据的纵向结构：

```text
市场状态带：阶段 / 风险预算 / 置信度 / 数据时间
核心判断：一句话结论 / 最强依据 / 最大风险 / 失效条件
趋势与宽度：主要指数脉络 / 涨跌分布 / 极端情绪
风格与主线：风格矩阵 / 行业扩散 / 主线生命周期
流动性与事件：成交变化 / 资金代理 / 风险日历与新闻
明日情景：偏强 / 基准 / 偏弱及各自触发条件
证据审计：数据源 / 缺失字段 / 降级原因
```

移动端顺序固定为：市场状态、最大风险、风险预算、主线、趋势宽度、三情景、证据审计。最大风险不折叠。

### 4.4 视觉表达

- 保留当前深色研究终端气质，但减少同权重卡片。
- 用一条横向“市场状态带”作为首屏视觉锚点，不使用多个巨大 KPI 数字争夺注意力。
- 涨跌分布继续使用现有分布图，但与历史分位或前一交易日对比；没有对比数据时明确标注“仅当日截面”。
- 板块不只列涨幅，增加扩散、持续、龙头依赖和退潮信号；字段缺失则显示缺口。
- 红色表示上涨/风险提示必须通过标签和文案区分，不能只靠颜色表达语义。

### 4.5 大盘验收标准

- 任一市场阶段都能追溯到至少两项支持证据和一项反证或缺口。
- 同一输入重复运行得到相同阶段、风险预算和失效条件。
- 行情过期时不展示进攻性风险预算。
- 只有当日截面时，不出现“持续增强”“风格切换完成”等跨期结论。
- 页面首屏不再把“买卖指导”作为大盘主体，而显示市场风险预算和状态变化条件。

## 5. 个股分析：投资备忘录与证据审计

### 5.1 页面必须回答的问题

个股首屏在 30 秒内回答：

1. 公司靠什么赚钱，当前经营质量如何。
2. 市场当前交易的核心预期是什么。
3. 当前价格和估值隐含了什么预期。
4. 最关键的多头证据、空头证据和核心分歧是什么。
5. 哪些数据缺失会阻止形成结论。
6. 在基准、乐观、悲观情景下，结论如何变化。
7. 什么条件下进入观察、行动、减仓或证伪。

### 5.2 新研究契约

新增 `StockResearchMemo`，由以下模块组成：

- `business_snapshot`：公司业务、收入/利润驱动、行业位置；数据不足时只显示已知事实。
- `quality_assessment`：营收、利润、ROE、毛利率、负债、经营现金流与利润匹配度的趋势和异常。
- `valuation_assessment`：PE、PB、PS、历史分位、行业对比；只拥有绝对 PE/PB 时不得输出“低估”。
- `expectation_gap`：市场预期、当前可验证事实、潜在催化、可能证伪项。
- `technical_structure`：趋势、支撑压力、量价、波动和关键失效位。
- `capital_and_event`：资金行为、新闻、公告和事件风险，区分事实与标题级规则判断。
- `portfolio_context`：是否持仓、成本位置、组合集中度影响和风险预算占用。
- `bull_case`、`base_case`、`bear_case`：各自的前提、观察指标、风险和失效条件，不强行给目标价。
- `research_verdict`：研究状态、置信度、核心矛盾、最强证据、最强反证和下一次复核动作。
- `evidence_audit`：每条核心结论对应数据源、日期、字段和缺口。

### 5.3 结论规则

- 若基本面、估值和公告三类中缺少两类，最高只能输出“技术性观察”，不能输出“投资逻辑成立”。
- 若只有绝对估值值而没有历史/行业参照，估值结论只能是“绝对估值水平”，不能称高估或低估。
- 若财务数据只有单期，不能声称趋势改善或恶化。
- 若公告只做标题规则识别，必须标注“标题级风险扫描，未代替原文复核”。
- 技术触发不能反向证明基本面质量；基本面良好也不能自动生成买点。
- TradingAgents 风格角色链保留为折叠式审议记录，不作为首屏主结构。

### 5.4 页面结构

```text
股票身份带：名称 / 代码 / 市场 / 价格 / 日期 / Provider
研究结论：状态 / 置信度 / 核心矛盾 / 最强证据 / 最强反证
投资备忘录：公司怎么赚钱 / 市场在交易什么 / 当前估值隐含什么
三情景：乐观 / 基准 / 悲观，各自前提与证伪
六类证据：经营质量 / 估值 / 技术 / 资金 / 事件 / 板块
组合上下文：持仓成本 / 风险暴露 / 集中度影响
研究行动：下一步需要验证什么 / 何时复核
交易计划：触发 / 仓位上限 / 止损 / 禁止动作
证据审计：来源 / 日期 / 原始字段 / 缺失与降级
完整角色审议：折叠展示
```

六维评分不再出现在页面标题旁，不用“机会评分 80/100”抢占主视觉。可以在证据区保留评分作为排序辅助，但必须同时显示证据和缺口。

### 5.5 个股验收标准

- 首屏明确展示核心矛盾、最强正反证、置信度和数据缺口。
- 三种情景的前提互不矛盾，并分别包含可验证指标与失效条件。
- 财务、估值、新闻或资金字段变化时，相关结论和置信度按规则变化。
- 缺少历史估值或行业对比时，页面不出现“低估”“估值安全垫”等无依据文案。
- 未持仓时不生成成本优势或组合减仓结论；已持仓时显示成本位置和组合风险影响。
- 原始证据能够从结论定位到来源和日期。

## 6. 架构与代码边界

不继续把新研究逻辑写入 `web.py`。实施时建立以下边界：

```text
providers / snapshot
  -> research.market_regime
  -> research.stock_memo
  -> view_models.market_view / stock_view
  -> webapp renderers
  -> web.py orchestration only
```

建议新增：

- `src/stock_ts/research/market_regime.py`
- `src/stock_ts/research/stock_memo.py`
- `src/stock_ts/research/evidence.py`
- `src/stock_ts/webapp/market_workspace.py`
- `src/stock_ts/webapp/stock_workspace.py`

`web.py` 仅负责获取已有报告、构造研究对象并调用渲染器。旧函数先保留兼容代理，待回归测试稳定后再删除，避免一次性重写 13,000 行入口文件。

## 7. 数据流与降级

1. Provider 返回标准行情、财务、估值、资金、新闻和公告字段。
2. 数据审计层为每个数据块生成来源、日期、完整度和限制。
3. 市场/个股研究模型只使用通过日期和完整度检查的数据。
4. 研究模型生成结构化结论，不生成 HTML。
5. View model 压缩内容并决定首屏、折叠和缺失状态。
6. Renderer 只负责语义化 HTML 与可访问性。

错误处理分三级：

- `blocked`：行情过期或关键身份/价格数据缺失，暂停结论。
- `degraded`：部分研究块缺失，只输出受限结论并降低置信度。
- `complete`：核心数据块满足研究要求，但仍显示来源和日期。

任何异常不得回退到 sample 并伪装真实结果。规则分析失败时显示该研究块失败，不吞掉异常后输出默认强结论。

## 8. 测试策略

### 8.1 研究模型测试

- 市场状态：进攻、轮动、震荡、防守、风险释放和数据暂停。
- 置信度：关键字段缺失、单日截面、过期行情和多源冲突。
- 个股备忘录：完整财务、单期财务、无估值参照、负现金流、高负债和事件风险。
- 三情景：前提、证据、失效条件完整且互不矛盾。
- 证据审计：核心结论都有来源、日期或明确缺口。

### 8.2 界面测试

- 验证语义区块、顺序、降级状态和移动端结构，不只验证关键词存在。
- 验证评分不再是首屏主结论。
- 验证缺数据时禁止出现无依据的“低估”“持续增强”和强行动文案。
- 验证从大盘主线到个股页的上下文参数仍可用。

### 8.3 回归测试

- 运行大盘、个股、数据质量、Provider、持仓上下文和 Web shell 的聚焦测试。
- 按 `docs/agent-ops/local-full-test-flow.md` 完成全项目测试并生成 Markdown 报告。
- 本地启动 Web，分别用完整快照、缺失财务快照和过期快照做浏览器烟测。

## 9. 实施顺序

1. 先为市场状态和个股研究备忘录写失败测试与领域契约。
2. 实现证据审计和数据降级规则。
3. 实现市场状态模型并替换每日大盘工作区。
4. 实现个股研究备忘录与三情景模型。
5. 拆出两个工作区渲染器，保留旧调用兼容层。
6. 完成聚焦测试、全项目测试和三种数据状态的本地 Web 验证。

## 10. 非目标

- 不做自动交易和券商下单。
- 不在本次接入付费研报全文、盈利预测一致预期或新的外部服务。
- 不承诺目标价或未来收益。
- 不重做持仓、热点机会、账户和通知页面。
- 不一次性拆分整个 `web.py`；只拆本次直接涉及的两个工作区。

## 11. 完成定义

完成后，用户应能在大盘页先确定风险环境，再进入个股页验证一条完整投资论点；任何结论都能看到支持证据、反证、数据缺口和失效条件。页面专业度来自研究纪律和可审计性，而不是更多卡片、更多分数或更强烈的买卖措辞。
