# StockTS 股票分析软件 Agent 设计指南

> 目标读者：后续接手本项目的 AI agent / 软件开发者 / 页面设计者。  
> 目标：让 agent 明白这个股票分析软件应该如何设计、如何分模块、页面应该怎么展示、用户应该如何操作，以及哪些设计不要再重复出现。  
> 当前原型参考：`reports/prototypes/stock-analysis-framework-prototype.html`。

## 1. 产品定位

StockTS 是一个面向 A 股个人投研的股票分析软件。它不是“指标大全”，也不是“自动荐股/自动交易系统”，而是帮助用户完成每日研究闭环：

1. 判断今天市场环境。
2. 找到强弱板块和短线情绪。
3. 生成值得观察的股票候选池。
4. 对单只股票做证据链分析。
5. 管理持仓风险和操作计划。
6. 形成每日复盘报告。
7. 通过消息渠道做提醒或发送。

所有交易相关内容必须使用“研究、观察、条件、风险、失效”表达，不能承诺收益，不能使用“必涨、稳赚、无风险”等措辞。

## 2. 核心设计原则

### 2.1 每个模块只做自己的事

页面左侧是模块导航。用户点击模块后，主区域只展示该模块相关内容和该模块自己的操作。

不要在每个模块顶部重复放全局区块，例如：

- 不要每个模块都显示同一个“交易日 / Provider / 样例数据 / 合规边界”大卡片。
- 不要每个模块都显示同一个“快速操作 / 个股计划”全局操作台。
- 不要让用户在所有模块里都看到同一组无关按钮。

如果一个信息必须全局可见，优先放在左侧导航、底部状态、设置页或当前模块内部的轻量提示里，不要占据每个模块首屏。

### 2.2 页面必须能操作

这个软件不是静态报告页。每个模块都应该有与该模块直接相关的操作入口，例如筛选、刷新、保存、添加、复制、发送 dry-run。

但操作必须是模块专属的：

- `智能选股` 里放筛选和排序。
- `个股分析` 里放股票输入、观察价、止损价、加入自选/持仓。
- `持仓风控` 里放新增、编辑、删除持仓。
- `每日复盘` 里放生成、复制、导出、发送。
- `消息自动化` 里放测试消息、dry-run、发送通道。

### 2.3 首页是导航，不是所有模块的集合

首页回答“我今天先看什么”。它可以展示市场脊柱、关键风险和快速入口，但不要堆满所有模块的细节。

首页应该告诉用户：

- 当前市场是否适合进攻、震荡还是防守。
- 今天优先看哪个板块或哪类风险。
- 下一步应该进入哪个模块。

### 2.4 模块内部遵循统一结构

每个模块建议采用以下结构：

1. 模块标题：清楚说明当前模块。
2. 一句话结论：回答本模块的核心问题。
3. 关键指标或表格：只展示和本模块相关的数据。
4. 模块专属操作：筛选、保存、添加、复制等。
5. 风险和失效条件：说明什么时候结论失效。
6. 数据质量提示：只在必要时出现，不要喧宾夺主。

### 2.5 不要变成“AI 生成感”的模板页

视觉上要像一个研究工作台，而不是普通后台管理系统。

当前方向：

- 左侧固定模块导航。
- 中间主内容区。
- 浅色研究纸张背景 + 深色交易终端卡片。
- 红色表示 A 股上涨/强势，绿色表示下跌/弱势。
- 深蓝用于重要决策卡片。
- 黄铜色用于提醒、样例数据、状态标签。

但设计要服务操作，不要为了装饰堆动画、渐变和无意义卡片。

2. A 股大盘
3. 板块主题
4. 涨跌停情绪
5. 智能选股
6. 个股分析

## 4. 模块设计

### 4.1 首页总览

核心问题：今天先看什么？

应该展示：

- 今日一句话。
- 市场脊柱：大盘环境、板块主线、短线情绪、候选机会、组合风险。
- 下一步判断：告诉用户应该先进入哪个模块。
- 关键指标卡：市场热度、上涨/下跌、涨停/跌停、数据可信度。
- 快速跳转按钮：个股分析、每日复盘、持仓风控等。

应该支持的操作：

- 点击进入关键模块。
- 根据 hash 或导航恢复当前模块。

不要做：

- 不要在首页放所有模块的完整表格。
- 不要把首页做成纯报告长文。

### 4.2 A 股大盘

核心问题：今天市场适合进攻、震荡还是防守？

应该展示：

- 市场状态：进攻 / 震荡 / 防守 / 退潮。
- 指数矩阵：上证、深成指、创业板、科创等。
- 市场广度：上涨家数、下跌家数、涨停数、跌停数。
- 成交额和缩放量状态。
- 风险闸门：跌破均线、成交不足、跌停扩散等。

应该支持的操作：

- 切换指数范围。
- 切换交易日。
- 刷新大盘数据。
- 查看风险闸门详情。

不要做：

- 不要在大盘模块推荐具体股票。
- 不要把大盘单日上涨直接解释为趋势突破。

### 4.3 板块主题

核心问题：今天主线在哪里，持续性如何？

应该展示：

- 板块热度榜。
- 主线确认矩阵：涨幅、广度、成交、涨停数、风险。
- 轮动提示：昨日强今日弱、低位补涨、高位退潮。
- 代表股和风险股。

应该支持的操作：

- 按涨幅、成交、广度、涨停数排序。
- 点击板块查看成分股。
- 将板块加入重点观察。
- 跳转到智能选股并自动带入板块筛选条件。

不要做：

- 不要只按涨幅判断主线。
- 不要展示全量板块列表而不突出主线。

### 4.4 涨跌停情绪

核心问题：短线赚钱效应和亏钱效应怎么样？

应该展示：

- 涨停数、跌停数、连板高度、炸板率。
- 涨停梯队。
- 跌停风险分布。
- 情绪周期：冰点、修复、高潮、分歧、退潮。

应该支持的操作：

- 切换涨停 / 跌停 / 连板 / 炸板视图。
- 按板块筛选涨跌停股票。
- 点击股票进入个股分析。
- 将强势股加入候选池或自选股。

不要做：

- 不要把涨停列表做成追板推荐。
- 如果数据源只能近似识别涨跌停，必须明确标注。

### 4.5 智能选股

核心问题：哪些股票值得观察？

应该展示：

- 筛选条件：板块、最低评分、趋势、量能、风险排除。
- 候选池表格：代码、名称、板块、综合分、入选理由、失效条件。
- 风险扣分和排除原因。

应该支持的操作：

- 按板块筛选。
- 按最低综合分筛选。
- 按综合分、趋势分、量能分排序。
- 将股票加入自选。
- 将股票加入每日复盘关注。
- 点击股票进入个股分析。

不要做：

- 不要使用确定性荐股口吻。
- 不要隐藏风险扣分。

### 4.6 个股分析

核心问题：这只股票为什么值得或不值得跟踪？

应该展示：

- 股票名称和代码。
- 核心结论：观察、谨慎观察、回避、持有复核等。
- 观察分和多角度评分。
- 技术结构：现价、压力、支撑、均线、量能。
- 新闻公告风险。
- 多角色观点：多头、空头、裁判。
- 操作计划：观察价、止损价、备注、禁止动作。

应该支持的操作：

- 输入股票代码或名称并刷新分析。
- 修改观察/压力价。
- 修改失效/止损价。
- 保存个股计划。
- 加入自选。
- 加入持仓。
- 复制个股研究摘要。

不要做：

- 不要只展示静态结论，没有可编辑计划。
- 不要把 AI 观点作为唯一依据。

### 4.7 持仓风控

核心问题：我的组合风险是否可承受？

应该展示：

- 组合健康度。
- 现金仓位。
- 前三大持仓集中度。
- 行业暴露。
- 持仓表格：代码、名称、仓位、浮盈亏、建议动作、止盈/止损。

应该支持的操作：

- 新增持仓。
- 编辑持仓。
- 删除持仓。
- 从交易流水生成持仓。
- 调整目标仓位。
- 将持仓加入每日复盘风险 Top 3。

不要做：

- 不要只展示盈亏，不展示集中度和行业暴露。
- 不要把持仓建议写成无条件买卖指令。

### 4.8 回测策略

核心问题：这个想法历史上是否有验证价值？

应该展示：

- 策略参数。
- 总收益、基准收益、最大回撤、胜率、交易次数。
- 资金曲线和回撤曲线。
- 交易明细。
- 回测局限说明。

应该支持的操作：

- 修改策略参数。
- 运行回测。
- 切换标的和时间区间。
- 导出交易明细。
- 将策略结果写入复盘。

不要做：

- 不要只展示收益，不展示回撤。
- 不要把历史回测解释为未来收益保证。

### 4.9 自选研究

核心问题：我的长期观察池如何管理？

应该展示：

- 自选分组。
- 股票、研究假设、提醒条件。
- 标签：行业、主题、风格、风险。
- 历史观察记录。

应该支持的操作：

- 添加自选。
- 修改研究假设。
- 修改提醒条件。
- 删除自选。
- 点击进入个股分析。
- 从智能选股或个股分析快速加入自选。

不要做：

- 不要让自选股只有代码，没有研究假设。
- 不要让提醒看起来像自动交易。

### 4.10 每日复盘

核心问题：今天发生了什么，明天看什么？

应该展示：

- 今日一句话。
- 最重要的三件事。
- 大盘与板块复盘。
- 涨跌停情绪复盘。
- 持仓风险 Top 3。
- 候选股 Top N。
- 重点个股计划。
- 明日观察计划。
- 免责声明。

应该支持的操作：

- 生成复盘。
- 复制 Markdown。
- 导出 HTML / Markdown。
- 发送 dry-run。
- 保存归档。

不要做：

- 不要每次打开页面就自动真实发送。
- 不要生成没有数据来源和免责声明的报告。

### 4.11 消息自动化

核心问题：如何安全地发送报告或提醒？

应该展示：

- 邮件、企业微信、飞书等渠道状态。
- 发送样式：完整、摘要、行动清单。
- dry-run 预览。
- 最近发送记录。

应该支持的操作：

- 发送测试消息。
- dry-run 今日复盘。
- dry-run 行动清单。
- 复制消息内容。
- 切换默认发送样式。

不要做：

- 不要在页面展示完整 token、Webhook、邮箱授权码。
- 不要默认真实发送。

### 4.12 系统设置

核心问题：数据和系统是否可信？

应该展示：

- 数据源矩阵：sample、Tencent、AKShare、Tushare、TDX snapshot、本地 CSV。
- 当前 Provider 状态。
- 缓存状态。
- 凭证配置状态：只显示 configured / missing。
- doctor 体检结果。

应该支持的操作：

- 刷新数据源状态。
- 清理缓存。
- 运行体检。
- 查看配置说明。
- 切换默认 Provider。

不要做：

- 不要展示真实密钥。
- 不要让数据源失败时页面静默显示旧数据。

## 5. 页面交互规范

### 5.1 导航

- 左侧导航固定显示 12 个模块。
- 当前模块要有明显高亮。
- URL hash 应该能恢复模块，例如 `#stock`、`#daily`。
- 移动端导航可以折叠或限制高度，不要占满首屏。

### 5.2 操作反馈

所有用户操作都应该有反馈：

- 保存成功：显示 toast 或模块内提示。
- 筛选无结果：显示空状态和下一步建议。
- 数据源失败：显示错误原因和修复建议。
- dry-run：明确说明“未真实发送”。

### 5.3 表单

表单必须贴近模块场景：

- 个股分析：股票、观察价、失效价、备注。
- 智能选股：板块、最低分、风险排除。
- 持仓风控：代码、名称、仓位、成本价、行业。
- 自选研究：分组、研究假设、提醒条件。
- 消息自动化：渠道、样式、dry-run。

不要做一个所有模块共用的大表单。

### 5.4 空状态

空状态要告诉用户下一步怎么做：

- 候选池为空：降低最低评分或切换板块。
- 持仓为空：上传持仓 CSV 或新增持仓。
- 自选为空：从智能选股或个股分析加入。
- 消息渠道未配置：去系统设置查看环境变量说明。

## 6. 数据与状态设计

前端页面不应该直接依赖外部 SDK 字段。建议后端或服务层输出稳定 ViewModel。

### 6.1 推荐 ViewModel

```text
HomeViewModel
MarketViewModel
SectorViewModel
SentimentViewModel
ScreenerViewModel
StockResearchViewModel
PortfolioRiskViewModel
BacktestViewModel
WatchlistViewModel
DailyReportViewModel
NotificationViewModel
SystemStatusViewModel
```

### 6.2 通用字段

每个 ViewModel 建议包含：

```text
module_id
module_title
summary
trade_date
provider
is_sample
is_stale
warnings
primary_actions
secondary_actions
data
```

### 6.3 不要重复全局块

`provider`、`trade_date`、`is_sample` 这些字段可以存在于 ViewModel 中，但页面呈现时不要每个模块都放大卡片重复展示。

建议呈现方式：

- 模块标题附近放一个轻量 badge。
- 系统设置页展示完整数据源矩阵。
- 报告里统一写数据来源和免责声明。

## 7. 前端实现建议

当前原型是单文件 HTML。后续工程化时可以拆成：

```text
src/stock_ts/webapp/
  shell.py              # 应用壳、左侧导航、基础布局
  view_models.py        # 页面展示模型
  components.py         # 卡片、表格、标签、按钮、空状态
  pages/
    home.py
    market.py
    sectors.py
    sentiment.py
    screener.py
    stock.py
    portfolio.py
    backtest.py
    watchlist.py
    daily.py
    notifications.py
    settings.py
  styles.py             # CSS 变量和视觉系统
  actions.py            # 页面动作与后端 workflow 对接
```

如果继续使用纯 HTML/JS 原型，至少要保持：

- 每个模块一个渲染函数。
- 用户操作集中在 action handler。
- 测试数据集中在 sample state。
- 不要在 HTML 字符串里到处复制同一段全局 UI。

## 8. 后端与工作流建议

后端保持现有分层思路：

```text
CLI / Web
  -> workflow orchestration
  -> analysis services
  -> provider adapters
  -> cache
  -> report renderer
  -> notification
```

模块和后端能力对应关系：

| 前端模块 | 后端服务 |
| --- | --- |
| 首页总览 | workflow 聚合各模块摘要 |
| A 股大盘 | market analysis service |
| 板块主题 | sector analysis service |
| 涨跌停情绪 | sentiment / limit-up service |
| 智能选股 | screener / candidate scoring service |
| 个股分析 | stock research / deep analysis / announcements |
| 持仓风控 | portfolio / portfolio_advice |
| 回测策略 | backtest service |
| 自选研究 | watchlist service |
| 每日复盘 | report / deep_report |
| 消息自动化 | notification |
| 系统设置 | config / doctor / data_sources |

## 9. 测试要求

### 9.1 前端原型测试

至少检查：

- 12 个模块导航存在。
- 不存在重复全局顶部区：`topbar`、`status-ribbon`、`operation-panel`。
- 每个模块只展示自己的内容。
- 智能选股有筛选入口。
- 持仓风控有新增持仓入口。
- 自选研究有添加自选入口。
- 每日复盘有复制 Markdown 入口。
- 消息自动化有 dry-run 入口。

### 9.2 后端测试

至少覆盖：

- sample provider 离线可运行。
- 真实 provider 失败时安全降级。
- 数据过期时页面有提示。
- 报告包含免责声明。
- 密钥不会出现在页面、日志、报告、错误信息中。
- 用户操作不会直接触发真实交易。

## 10. 明确不要做的事

后续 agent 不要做以下事情：

1. 不要把每个模块顶部都塞同一块全局状态卡。
2. 不要把每个模块顶部都塞同一块全局操作台。
3. 不要把股票软件做成纯静态展示页。
4. 不要把所有操作都放在首页。
5. 不要在大盘模块推荐个股。
6. 不要把智能选股写成确定性荐股。
7. 不要隐藏样例数据、降级数据、过期数据状态。
8. 不要在页面展示真实 token 或 webhook。
9. 不要默认真实发送消息，必须先 dry-run。
10. 不要输出收益承诺。

- 删除了重复的顶部交易信息区。
- 删除了重复的全局数据质量状态条。
- 删除了重复的全局“快速操作 / 个股计划”操作台。
- 保留左侧 12 模块导航。
- 保留首页市场脊柱。
- 保留模块内部自己的交互能力。

当前原型辅助检查脚本：

```text
reports/prototypes/check_interactive_prototype.py
```

这个脚本应该验证：

- 不存在重复全局块。
- 模块内部交互能力仍然存在。
- HTML 能正常解析。

## 12. 推荐下一步开发顺序

### 第一阶段：把原型变成真实 Web 壳

1. 抽出左侧导航和主内容布局。
2. 每个模块拆成独立渲染函数或组件。
3. 接入现有 sample workflow。
4. 保留当前视觉方向。
5. 加前端结构测试。

### 第二阶段：模块内操作落地

1. 个股分析：输入股票、保存观察计划。
2. 智能选股：筛选、排序、加入自选。
3. 持仓风控：新增、编辑、删除持仓。
4. 每日复盘：生成、复制、导出。
5. 消息自动化：dry-run 和测试发送。

### 第三阶段：数据质量和自动化

1. 每个模块接入真实数据质量提示。
2. 数据源失败显示可读错误。
3. 报告归档。
4. 定时任务示例。
5. AI 增强研报，但不能阻断规则分析。

## 14. 一句话总结

StockTS 应该被设计成一个“模块清晰、能操作、能复盘、能解释风险”的 A 股研究工作台，而不是一个所有模块都重复全局卡片的静态报告页面。

## 15. 还需要补强的设计约束

本节是对前面设计指南的补充，用于降低后续 agent 的自由发挥空间，避免做出“看起来完整但不可落地”的页面。

### 15.1 目标用户与使用场景

后续设计必须围绕以下用户场景，不要抽象地做“金融仪表盘”。

| 用户场景 | 用户问题 | 应进入模块 | 关键操作 |
| --- | --- | --- | --- |
| 收盘后复盘 | 今天市场怎么样，明天先看什么？ | 首页总览、每日复盘 | 生成复盘、复制报告、发送 dry-run |
| 盘前准备 | 今天有哪些方向值得观察？ | A 股大盘、板块主题、智能选股 | 筛选候选、加入自选、生成观察计划 |
| 单股研究 | 这只股票是否值得继续跟踪？ | 个股分析 | 输入股票、保存观察价、保存失效价、加入自选 |
| 持仓检查 | 我的组合风险是否过高？ | 持仓风控 | 新增/编辑持仓、调整目标仓位、写入复盘 |
| 策略验证 | 某个规则历史上是否有用？ | 回测策略 | 修改参数、运行回测、导出结果 |
| 通知发送 | 把报告发给自己或团队 | 消息自动化 | 发送测试消息、dry-run、复制消息 |
| 系统排查 | 数据为什么不可信或没有更新？ | 系统设置 | 查看 Provider、运行 doctor、清缓存 |

### 15.2 用户路径优先级

开发时优先保证以下 4 条路径能跑通。

#### 路径 A：每日复盘闭环

```text
首页总览
  -> A 股大盘
  -> 板块主题
  -> 智能选股
  -> 持仓风控
  -> 每日复盘
  -> 消息自动化 dry-run
```

验收：用户能在不理解内部实现的情况下，从首页进入复盘，生成一份带免责声明的 Markdown 报告。

#### 路径 B：单股研究闭环

```text
个股分析
  -> 输入股票
  -> 查看证据链
  -> 修改观察价/失效价
  -> 保存计划
  -> 加入自选
  -> 写入每日复盘
```

验收：用户能围绕一只股票保存可复核的条件化计划。

#### 路径 C：持仓风控闭环

```text
持仓风控
  -> 新增/编辑持仓
  -> 查看集中度和行业暴露
  -> 设置目标仓位/止损止盈
  -> 写入持仓风险 Top 3
```

验收：用户能知道“组合最大风险是什么”，而不是只看到盈亏。

#### 路径 D：候选池到自选闭环

```text
板块主题
  -> 进入智能选股
  -> 按板块/评分筛选
  -> 查看入选理由和失效条件
  -> 加入自选研究
```

验收：用户能把短期机会沉淀成可跟踪的自选研究对象。

## 20. 错误、加载和空状态规范

### 20.1 加载状态

- 刷新行情时，模块内按钮显示“刷新中”。
- 长耗时任务不要阻塞整个页面，只阻塞当前模块。
- 失败后恢复按钮可点击状态。

### 20.2 错误状态

错误提示必须包含：

1. 哪个数据源失败。
2. 当前是否用了降级数据。
3. 用户下一步能做什么。

示例：

```text
AKShare 板块数据获取失败，当前显示上一交易日缓存。你可以点击“刷新数据源”重试，或切换到 TDX snapshot。
```

### 20.3 空状态

空状态不要只写“暂无数据”。必须给操作建议。

示例：

```text
当前筛选条件下没有候选股。可以降低最低综合分、切换板块，或关闭高风险排除。
```

## 21. 合规文案规范

所有涉及交易动作的模块必须使用条件句。

推荐写法：

- “若放量站稳 13.40，则提高观察优先级。”
- “若跌破 12.15，则降级观察。”
- “当前只做研究排序，不构成投资建议。”

禁止写法：

- “明天必涨。”
- “建议全仓买入。”
- “稳赚机会。”
- “无风险套利。”
- “跟着买就行。”

## 22. 最小可交付版本定义

如果后续 agent 要把设计落成真实代码，MVP 不要贪多。最小可交付版本只需要保证：

1. 左侧 12 模块导航可切换。
2. 没有重复全局大卡片。
3. 首页展示市场脊柱和下一步入口。
4. A 股大盘、板块主题、智能选股、个股分析、持仓风控、每日复盘至少能使用 sample 数据渲染。
5. 智能选股能筛选候选池。
6. 个股分析能保存观察价/失效价到页面状态。
7. 持仓风控能新增一条模拟持仓。
8. 每日复盘能生成并复制 Markdown。
9. 消息自动化能 dry-run，不真实发送。
10. 测试能证明以上功能存在。

MVP 不需要：

- 真实券商交易。
- 自动下单。
- 复杂权限系统。
- 全量历史数据库。
- AI 必须在线。

## 24. 研究终端密度与渐进展开规范

当前真实 Web 工作台采用“行动闸门 -> 模块结论 -> 研究前排 -> 完整审计记录”的阅读顺序。后续 agent 优化页面时，不得重新把所有状态、候选和持仓记录铺成同权卡片流。

### 24.1 全局 Research Tape

- 全局状态区只保留一个 Research Tape，不再使用六个同权状态卡。
- 动作闸门必须是视觉主项；数据状态和交易日是核心项；行情日期、证据覆盖和来源是次级项。
- 数据详情必须直接链接到数据中台，不能只显示不可操作的状态文字。
- stale、blocked 或高风险状态必须继续压制当前价格动作、主动排序和无依据的交易结论。

### 24.2 密集模块的前排规则

- 机会研究常驻前 6 只候选，其余候选进入 `查看其余 N 只候选`。
- 持仓处置常驻前 5 项，持仓边界常驻前 4 项，其余记录分别进入明确计数的展开区。
- 渐进展开只改变阅读密度，不得截断模型数据；股票链接、支持证据、最大反证、触发、失效条件和禁止动作必须完整保留在 HTML。
- 展开区默认关闭，使用原生 `details/summary`，并提供清晰的键盘焦点样式。

### 24.3 响应式验收口径

- 桌面宽屏保留完整 Research Tape，机会页风险登记表可在研究列表旁吸附。
- 390px 移动端隐藏品牌副标题和 Research Tape 次级字段，搜索框保持单行，导航项宽度控制在 112px。
- 移动端首屏必须出现当前模块标题或主结论，不能被全局状态区完全占满。
- 所有目标视口必须满足 `document.documentElement.scrollWidth <= window.innerWidth`，禁止横向溢出。

### 24.4 视觉取舍

研究终端的识别点只集中在 Research Tape：深海军蓝、风险酒红、象牙白和研究金构成状态语义。其余卡片保持安静、低装饰和高可读性，不新增无业务意义的渐变、徽章或动画。桌面和移动端都优先保证“先看到能否行动，再决定是否展开证据”。

## 25. 数据恢复指挥台规范

数据中台不是静态数据字典，也不是把所有采集字段平铺出来的技术表格。它的核心问题是：当前研究为什么被阻断，应该先恢复什么，恢复后如何证明可以重新形成结论。

### 25.1 固定阅读顺序

1. 数据就绪闸门：明确当前允许正常研究、降级研究还是停止强结论。
2. 恢复运行轨道：只展示阻断和需复核的数据域，并按真实依赖顺序排列。
3. 模块影响面：说明缺口会影响每日大盘、我的持仓、个股分析还是热点机会。
4. 完整来源账本：保留来源、时间、覆盖、缺口和研究影响，默认折叠。

### 25.2 恢复顺序

- 先恢复大盘行情和 K 线等上游价格输入。
- 再恢复候选池和资金面等筛选与承接证据。
- 然后恢复新闻、公告和基本面等事件与经营证据。
- 最后重新执行全链路校验，把它作为验收步骤，而不是可优先修复的数据源。
- 同一层级内，阻断项先于一般复核项；不得用综合分掩盖具体缺口。

### 25.3 内容边界

- 数据状态沿用既有 freshness、missing 和 blocked 规则，页面不得自行把滞后数据标记为可用。
- 每个恢复步骤必须同时写明当前缺口、业务后果和复核证据。
- 下一步指引只显示首要动作，长缺口清单需要收敛；完整文本仍保留在恢复卡和来源账本。
- 来源账本可以展示安全的 provider、文件产物和覆盖信息，但不得展示凭证、内部地址或密钥。
- 手动刷新仍是唯一写操作，不提供没有执行证据的“一键修复成功”。

### 25.4 响应式规范

- 桌面端使用闸门 + 恢复轨道 + 影响矩阵的指挥台结构。
- 390px 移动端全部切为单列，恢复顺序不变。
- 来源表在移动端转换为带字段标签的审计卡片，不允许依赖横向滚动阅读。
- 来源账本默认关闭，展开后仍必须满足页面无横向溢出。
